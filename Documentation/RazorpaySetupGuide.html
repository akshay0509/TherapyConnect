1

What you are setting up

When a booking is created (and payments are toggled on in your profile), the app creates a Razorpay Payment Link and Razorpay sends it to the client by SMS and email. When the client pays, Razorpay calls the app's webhook, the payment is recorded against the appointment, and the appointment is auto-confirmed.

To make this work you need exactly three values from the Razorpay dashboard, captured as environment variables on the Oracle VM:

Environment variableWhat it isWhere it comes from
RAZORPAY_KEY_ID Public API key id (starts with rzp_test_ or rzp_live_) Dashboard → Account & Settings → API Keys
RAZORPAY_KEY_SECRET Secret paired with the key id — shown only once at generation Same screen, at key generation time
RAZORPAY_WEBHOOK_SECRET Secret you invent yourself; used to verify webhook authenticity (HMAC signature) You type it into Dashboard → Webhooks when creating the webhook, and put the same value in .env
Kill switch: if these variables are blank or missing, payments are silently disabled across the whole app — bookings work exactly as before. There is also the per-profile toggle ("Turn on payments") on the Therapist Profile page.
2

Create the account

Step 1Sign up

  1. Go to https://dashboard.razorpay.com/signup
  2. Sign up with your email and phone number
  3. Choose business type — for a solo practice, Individual / Proprietorship is the usual fit

Step 2KYC (needed for live payments only)

You can integrate and test immediately in Test Mode without KYC. To accept real money you must complete KYC activation:

  • PAN (personal PAN is fine for individual)
  • Bank account details (where settlements land, typically T+2 days)
  • Address proof / Aadhaar
  • Business description — describe it as therapy/counselling services

Approval usually takes a few working days. Do all your testing in Test Mode while you wait.

3

Generate API keys

Step 3Test keys first

  1. In the dashboard, make sure the mode switch (top of the page) is on Test Mode
  2. Go to Account & Settings → API Keys → Generate Test Key
  3. Copy both values immediately:
    • Key Id → becomes RAZORPAY_KEY_ID (looks like rzp_test_Ab12Cd34Ef56Gh)
    • Key Secret → becomes RAZORPAY_KEY_SECRET
The secret is shown only once. If you lose it, you must regenerate the key pair (which invalidates the old one). Store it straight into the .env file.
4

Configure the webhook

Step 4Register the webhook URL

  1. Dashboard → Account & Settings → Webhooks → Add New Webhook
  2. Webhook URL:
    http://<your-vm-public-ip-or-domain>:8091/appointment/webhook/razorpay
    (the gateway strips /appointment and routes to AppointmentService)
  3. Secret: invent a long random string (e.g. output of openssl rand -hex 24) — this exact value goes into RAZORPAY_WEBHOOK_SECRET
  4. Active events — tick these four:
    • payment_link.paid — marks the appointment paid and auto-confirms it
    • payment_link.cancelled
    • payment_link.expired
    • payment.failed — records the failure reason (client can retry the same link)
Reachability: Razorpay must be able to reach that URL from the internet. On the Oracle VM this means port 8091 must be open in BOTH the Oracle Cloud Security List (VCN) and the VM firewall (iptables/ufw). Razorpay requires the webhook to respond within a few seconds — the app answers immediately after the signature check, so this is fine. For production, put this behind HTTPS (e.g. nginx + Let's Encrypt on a domain) — Razorpay strongly recommends HTTPS webhook URLs and live mode may reject plain HTTP.
5

Capture the environment variables

Step 5Add to .env on the Oracle VM

Append to ~/TherapyConnect/.env:

# ── Razorpay (Test Mode) ─────────────────────────────
RAZORPAY_KEY_ID=rzp_test_XXXXXXXXXXXXXX
RAZORPAY_KEY_SECRET=XXXXXXXXXXXXXXXXXXXXXXXX
RAZORPAY_WEBHOOK_SECRET=XXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXX

Only the appointment service reads these (wired in docker-compose.yml). Recreate it after editing:

cd ~/TherapyConnect
docker compose up -d appointment
.env holds secrets — make sure it is in .gitignore and never committed.
6

Test the flow (Test Mode)

Step 6End-to-end checklist

  1. Turn payments on from the Therapist Profile page (this also creates the payment projection)
  2. Make sure the client you book for has a valid email/phone (edit the client once if they were created before this feature — that refreshes the contact projection)
  3. Book an appointment → the booking panel should show the payment link
  4. Open the link yourself and pay with test credentials:
    • Card: 4111 1111 1111 1111, any future expiry, any CVV
    • UPI: success@razorpay (or failure@razorpay to test a failed payment)
  5. Within a few seconds the appointment should flip to CONFIRMED and the update panel should show ✓ Paid
  6. Also test: cancel an unpaid appointment (link should die), reschedule an unpaid appointment (new link should be issued)
In Test Mode, Razorpay generally does not deliver real SMS/email to the client — verify delivery only after going live. The link itself and the webhook flow work fully in Test Mode.
7

Going live

Step 7Switch to live keys

  1. Wait for KYC activation approval
  2. Switch dashboard to Live Mode → generate Live API keys
  3. Create the webhook again in Live Mode (webhooks are per-mode) with the same URL and a fresh secret
  4. Replace the three values in .env with the live ones and docker compose up -d appointment
  5. Make one small real booking/payment to verify, then refund it from the dashboard if needed
8

Fees and operational notes

  • Fees: ~2% + 18% GST on the fee per domestic transaction (≈ ₹30.7 on a ₹1,300 session). No setup or monthly fee.
  • Settlements: money lands in your bank account on a T+2 working-day cycle by default.
  • Refunds: manual via the Razorpay dashboard (Payments → find by appointment id in notes → Refund). The app flags "manual refund may be required" in the appointment-service logs if a payment lands on a cancelled appointment.
  • Key rotation: if a secret ever leaks, regenerate keys in the dashboard, update .env, restart the appointment service.
  • Monitoring: the dashboard's Webhooks page shows delivery attempts and failures — first place to look if payments stop confirming appointments.
