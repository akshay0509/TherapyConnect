Technical Debt & Risks

An honest, evidence-based assessment of gaps to close before production, ranked by area. Each item cites what was observed in the source.

This is a capability-rich system with thoughtful patterns (outbox, CQRS, circuit breakers, field encryption, race-safe booking). The items below are about hardening, not redesign.

Security

1 · Refresh tokens drop roles High

AuthController.refresh mints the new JWT with Set.of() for authorities (roles aren't stored on the REFRESH_TOKENS row — noted in a code comment). Any role-based authorization added later would silently fail for refreshed sessions.

Fix: persist roles on the refresh row (or re-fetch from UserService on refresh) and re-populate the claim.

2 · Broad Kafka deserialization trust Medium

Most consumers set spring.json.trusted.packages=*. Acceptable on a closed internal bus, but risky if the bus is ever shared. Fix: narrow to com.org.events.* (as UserService already does for login events).

3 · Secret management High

The RSA private signing key is loaded from classpath:keys/gateway-private.pem, and DB/Google/AES secrets come from per-service .env files. Fix: move to a secret manager/KMS; never bake keys into JARs; rotate the notes key with envelope encryption.

Other security notes Low/Med

/user/validate-user is permitAll (intended gateway-only — restrict by network/mTLS). The user JWT is propagated to downstream services rather than using short-lived service tokens. Consider audience/issuer validation enforcement on resource servers.

Data & schema

4 · No migration tooling High

Every service runs spring.jpa.hibernate.ddl-auto=update. This is convenient in dev but unsafe in production — it can't drop/rename safely, offers no review trail, and risks divergence. Fix: adopt Flyway or Liquibase, switch to validate, and baseline the current schema.

5 · One DB credential set Medium

All services share DB_URL/USERNAME/PASSWORD. "Database per service" is logical only; nothing prevents cross-table reads, undermining isolation. Fix: separate schemas/databases with per-service users and least-privilege grants.

Reliability & messaging

6 · No dead-letter strategy Medium

The outbox guarantees at-least-once publication, and consumers are idempotent, but there is no platform-wide DLQ + retry/backoff. A consistently failing message blocks progress (the outbox scheduler breaks the batch on first failure) or is dropped after error-handling deserialization. Analytics' config hints at DLT but it isn't fully realized. Fix: add a Spring Kafka DefaultErrorHandler with retries + DLT topics, and alerting on outbox backlog / DLT depth.

Outbox single-instance assumption Low

The 2s poller has no locking; running multiple instances of a producing service could double-publish (consumers absorb it, but it's wasteful). Fix: use SELECT … FOR UPDATE SKIP LOCKED or a leader/sharded poller when scaling horizontally.

Operations & tooling

7 · No containers, CI, or health probes Medium

No Dockerfiles, Compose, K8s manifests, or CI pipeline; Spring Boot Actuator isn't wired, so there are no /health//ready endpoints for orchestration. Fix: add Actuator, a Compose file for local infra, per-service images, and a CI build running mvn verify.

Code quality

8 · Cleanups Low

  • Debug System.out.println calls remain in hot paths (e.g. LoginEventProducer, TherapistController.createClient).
  • Some controllers swallow exceptions into "failed" 200/500 strings (e.g. appointment booking on JsonProcessingException) instead of structured errors.
  • Commented-out dead code blocks (e.g. manual JSON serialization in LoginEventProducer).
  • Test classes are mostly default contextLoads stubs; there's little behavioral coverage of booking/availability logic. A handful of test classes also sit under a mismatched com.org.clientService package in other services.
  • Naming inconsistencies across services (e.g. OUTBOX_EVENT id prefixes, therapistService log level reused in appointment-service config).

Scalability considerations

  • Read models grow unbounded — projections and audit/outbox tables need archival/retention policies.
  • Analytics recomputation is event-sourced into rollups; there's no rebuild/replay tool if a projection is corrupted (mitigated by earliest offset but not formalized).
  • Single-tenant assumptions — calendar/email use one therapist identity from env (THERAPIST_EMAIL, shared Google creds); true multi-therapist Google integration would need per-therapist OAuth.
