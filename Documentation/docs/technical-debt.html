Technical Debt & Risks

An honest, evidence-based assessment of gaps to close before production, ranked by area. Each item cites what was observed in the source.

This is a capability-rich system with thoughtful patterns (outbox, CQRS, circuit breakers, field encryption, race-safe booking). The items below are about hardening, not redesign.

✅ Resolved since the first review
  • Refresh tokens now preserve roles (a roles column was added) — refreshed JWTs carry full authorities.
  • Signing key is env-only — the gateway loads it from ${JWT_PRIVATE_KEY} (a mounted volume), with no classpath default baked into the image.
  • Kafka trusted.packages=* is moot on the outbox topics — type headers are off and nothing is resolved from headers (login pipeline aside).
  • Docker/Compose + Actuator health now exist — full docker-compose.yml, per-service Dockerfiles, healthchecks, and a Portainer UI.
  • Databases are now physically separate (one per service) instead of co-mingled.
  • Internal endpoints are denied at the gateway (/*/internal/**).
  • Login rate limiting is now per-client-IP — the old global Resilience4j bucket (one source could exhaust it and lock out all users) was replaced by a per-IP limiter keyed on the un-spoofable nginx X-Real-IP.
  • TLS at the edge — the gateway is bound to loopback behind a host nginx that terminates HTTPS, so the API (payments, webhooks, SameSite=None cookie) runs over TLS.

Security

2 · Single static admin credential Medium

The admin console authenticates against one ADMIN_USERNAME/ADMIN_PASSWORD env pair (constant-time compared, rate-limited 5/min, 8-hour token — good). But it's a single shared secret with no MFA, per-admin identity, or audit of admin actions beyond logs. Fix: per-admin accounts, MFA, and an admin-action audit trail before exposing the console beyond a trusted network.

4 · Secrets & internal trust Medium

The gateway signing key is now env/volume-only (good), but DB/Google/AES/Razorpay/admin secrets still live in a root .env. /user/validate-user is permitAll on the user service (denied at the gateway, but reachable service-side) and the user JWT is propagated downstream rather than using short-lived service tokens. Fix: secret manager/KMS; restrict validate-user by network/mTLS; consider service tokens + issuer/audience enforcement.

Data & schema

4 · No migration tooling High

Every service runs spring.jpa.hibernate.ddl-auto=update. This is convenient in dev but unsafe in production — it can't drop/rename safely, offers no review trail, and risks divergence. Fix: adopt Flyway or Liquibase, switch to validate, and baseline the current schema.

3 · One DB credential set Medium

Databases are now physically separate (one per service in the shared Postgres instance), but all services connect with the same DB_USERNAME/DB_PASSWORD, so any service could still reach another's database. Fix: per-service DB users with grants scoped to their own database.

Reliability & messaging

6 · No dead-letter strategy Medium

The outbox guarantees at-least-once publication, and consumers are idempotent, but there is no platform-wide DLQ + retry/backoff. A consistently failing message blocks progress (the outbox scheduler breaks the batch on first failure) or is dropped after error-handling deserialization. Analytics' config hints at DLT but it isn't fully realized. Fix: add a Spring Kafka DefaultErrorHandler with retries + DLT topics, and alerting on outbox backlog / DLT depth.

Outbox single-instance assumption Low

The 2s poller has no locking; running multiple instances of a producing service could double-publish (consumers absorb it, but it's wasteful). Fix: use SELECT … FOR UPDATE SKIP LOCKED or a leader/sharded poller when scaling horizontally.

Payments

6 · Refunds & edge cases are manual Medium

The payment flow is defensively built (never blocks booking, idempotent webhook, best-effort link cancel). But refunds are entirely manual via the Razorpay dashboard, and a payment that lands on an already-cancelled appointment is only logged ("manual refund may be required"), not reconciled. Fix: an automated refund path (or at least an alert/queue) for over-payments, and a reconciliation report of PAID-vs-appointment-status mismatches.

Operations & tooling

7 · CI & broker redundancy Medium

Docker Compose, per-service images, healthchecks, Actuator, and TLS (host nginx → loopback gateway) are now in place. Still open: no CI pipeline (nothing runs mvn verify on push), and Kafka runs a single node at replication factor 1 (no redundancy — a broker loss is data loss). Fix: add CI, and move to a multi-broker cluster with replication ≥ 3 for the durable topics.

Code quality

8 · Cleanups Low

  • Debug System.out.println calls remain in hot paths (e.g. LoginEventProducer, TherapistController.createClient).
  • Some controllers swallow exceptions into "failed" 200/500 strings (e.g. appointment booking on JsonProcessingException) instead of structured errors.
  • Commented-out dead code blocks (e.g. manual JSON serialization in LoginEventProducer).
  • Test classes are mostly default contextLoads stubs; there's little behavioral coverage of booking/availability logic. A handful of test classes also sit under a mismatched com.org.clientService package in other services.
  • Naming inconsistencies across services (e.g. OUTBOX_EVENT id prefixes, therapistService log level reused in appointment-service config).

Scalability considerations

  • Read models grow unbounded — projections and audit/outbox tables need archival/retention policies.
  • Analytics recomputation is event-sourced into rollups; there's no rebuild/replay tool if a projection is corrupted (mitigated by earliest offset but not formalized).
  • Single-tenant assumptions — calendar/email use one therapist identity from env (THERAPIST_EMAIL, shared Google creds); true multi-therapist Google integration would need per-therapist OAuth.
