Local Development Setup — the dev Spring Profile

Run any service from STS with a single environment variable. Added 2026-07-05.

1. What changed

Each Spring service now has an application-dev.properties next to its main application.properties. It contains every value that used to be supplied through per-service environment variables in the STS run configuration: local database URL and credentials, local Kafka, key file locations, dev admin credentials, and the frontend URL.

The old way (a dozen env vars per run configuration) is replaced by exactly one:

SPRING_PROFILES_ACTIVE=dev

2. How SPRING_PROFILES_ACTIVE works

Spring Boot always loads application.properties. When a profile named dev is active, it additionally loads application-dev.properties from the same location — and profile-specific files win whenever both define the same key.

The main properties use placeholders like spring.datasource.url=${DB_URL}. Without the env var and without a profile, startup fails ("could not resolve placeholder"). With the dev profile active, application-dev.properties defines spring.datasource.url=jdbc:postgresql://localhost:5432/... directly, which overrides the placeholder line entirely — the env var is never needed.

Full precedence order (highest wins) for the sources used in this project:

PrioritySourceUsed by
1 (highest)OS environment variablesVM via docker-compose environment:; optional local overrides (e.g. real Gmail creds for one test run)
2application-dev.properties (profile file, only when active)Local STS runs
3 (lowest)application.propertiesEverywhere — shared config, routes, resilience, Kafka serialization

This is why production is untouched: the VM never sets SPRING_PROFILES_ACTIVE, so the dev file is inert there — and even if it were somehow activated, the compose environment variables sit at higher precedence and would still win.

3. One-time machine setup

  1. Generate a dev-only JWT keypair at D:\keys (Git Bash): 
    mkdir -p /d/keys && openssl genpkey -algorithm RSA -pkeyopt rsa_keygen_bits:2048 -out /d/keys/gateway-private.pem && openssl rsa -pubout -in /d/keys/gateway-private.pem -out /d/keys/gateway-public.pem
    This must be a different keypair from the one on the Oracle VM. Locally issued tokens are then valid only against locally running services — dev and prod tokens can never cross over.
  2. Postgres credentials are pre-filled — they were copied from the STS launch configurations (Jul 2026), so they match the existing local databases: user, therapist, client, appointment, notification, analytics, and (gateway only) the default postgres database. Nothing to do unless the local postgres login changes.
  3. Delete the old key from the source tree (C5 hygiene): GatewayService/src/main/resources/keys/gateway-private.pem, then mvn clean. Nothing references that location any more.

4. Per-service STS run configuration

  1. Open the run configuration → Environment tab.
  2. Remove all previously configured variables (DB_URL, KAFKA_BOOTSTRAP_SERVERS, JWT_PUBLIC_KEY, …).
  3. Add the single variable SPRING_PROFILES_ACTIVE = dev.

Start order for a full local stack: local postgres and kafka running → DiscoveryService → GatewayService → whichever services you need (they register to localhost:8761 by default).

5. Key layout in dev

ServiceReads from D:\keysPurpose
GatewayServicegateway-private.pem + gateway-public.pemSigns and validates JWTs
User / Therapist / Client / Appointment / Analyticsgateway-public.pem onlyValidate JWTs
NotificationService—No JWT validation (Kafka consumer only)
DiscoveryService—No security config, no dev file needed
Never place a private key under any service's src/main/resources — Maven copies resources into the JAR, which is exactly the exposure the C5 fix removed. Keys live outside the repo: D:\keys on dev, ~/TherapyConnect/keys (mounted) on the VM.

6. Features that are off locally (by design)

FeatureDev behaviorTo test it locally
Razorpay payment linksKeys empty → isConfigured() false → whole payment flow is a clean no-opSet TEST-mode RAZORPAY_* env vars in AppointmentService's run config
Password-reset / username emailsGmail creds empty → fails only at send timeSet GOOGLE_* + GMAIL_FROM env vars in UserService's run config
Google Calendar invitesCreds empty → fails only at API-call timeSet GOOGLE_* + THERAPIST_EMAIL env vars in NotificationService's run config

Env vars set in a run configuration override the dev profile's empty defaults (priority 1 beats priority 2), so enabling one of these for a session requires no file edits.

7. Frontend against the local stack

# auth-app/.env.local  (gitignored)
VITE_API_BASE_URL=http://localhost:8091

Then npm run dev — the gateway's CORS config already allows http://localhost:5173.

Summary: one env var per STS run config, one keypair at D:\keys, one credentials find-and-replace. The dev admin login is admin / admin-dev-only. Production configuration is entirely unaffected — the VM's compose env vars outrank everything in these files.
