API Reference

Every REST endpoint, grouped by service. Paths shown are the external gateway paths the SPA calls; the gateway strips the leading prefix before forwarding. Expand any endpoint for request/response detail and side effects.

Conventions

Public paths: /auth/**, /user/create-user, /user/forgot-username, /user/forgot-password, /user/reset-password, and the Razorpay /webhook/**. /admin/** requires ADMIN authority; everything else needs a valid Authorization: Bearer <jwt>. Service-internal routes (/*/internal/**) are denied at the gateway and only reachable service-to-service. The therapistId / userId are read from JWT claims server-side via SecurityUtils — clients never send them in the body (a field marked “from JWT” is populated by the server). The gateway routes by prefix: /therapist/* → therapist-service, /appointment/* → appointment-service, /client/* → client-service, /user/* → user-service, /analytics/* → analytics-service.

Authentication gateway · /auth

POST/auth/loginAuthenticate & issue tokens

Auth: public · rate-limited to 10 attempts / 60s per client IP (custom LoginRateLimiter, HTTP 429 when exceeded).

Request

AuthRequest{ "username": "jdoe", "password": "••••••••" }

Responses

  • 200 — { "token": "<jwt>" } plus Set-Cookie: refresh_token (httpOnly, Secure, SameSite=None, path /auth/refresh, 7-day).
  • 401 — { "failureReason": "INVALID_CREDENTIALS | ACCOUNT_LOCKED | ACCOUNT_DISABLED | SERVICE_UNAVAILABLE" }
  • 429 — { "error": "Too many login attempts…" }

Flow & side effects: validates via UserServiceProxy.validateUser (circuit-broken); publishes auth-login-success/auth-login-failure to Kafka; resolves therapistId from TherapistService; signs a 15-min RS256 JWT; persists a REFRESH_TOKENS row (deleting any prior tokens for the user).

POST/auth/admin-loginAdmin console login

Auth: public · rate-limited to 5 attempts / 60s per client IP (LoginRateLimiter, stricter than user login since there is no account lockout).

Request: { "username": "…", "password": "…" } — checked against the ADMIN_USERNAME/ADMIN_PASSWORD env credentials with a constant-time comparison.

Responses: 200 { "token": "<jwt>" } — an 8-hour JWT with authorities:["ADMIN"] and userId:"admin" (no refresh token) · 401 invalid · 429 rate-limited.

POST/auth/refreshRotate refresh token, issue new access token

Auth: refresh_token cookie · Request: none (cookie only).

Responses: 200 { "token": "<jwt>" } + new rotated cookie · 401 { "error": "No refresh token" | "Invalid or expired refresh token" }.

Side effects: deletes the presented token and issues a fresh one (rotation). Now preserves the user's roles (persisted on the refresh-token row) and back-fills therapistId if it was null at login — so refreshed tokens carry full authorities. The cookie is SameSite=None; Secure for the cross-site SPA.

POST/auth/logoutRevoke refresh token

Auth: refresh cookie (optional) · Response: 200 always, with a cleared cookie (maxAge=0).

Side effects: deletes the user’s refresh tokens (revocation) if the cookie is present.

POST/auth/forgot-passwordRequest a reset email

Auth: public · Request: { "email": "user@example.com" }

Response: 200 { "message": "If the email exists, reset instructions will be sent." } — generic by design (no account enumeration). Proxies to UserService.

POST/auth/reset-passwordComplete a password reset

Auth: public (token-scoped) · Request: { "token": "…", "newPassword": "…" }

Response: 200 { "message": "Password reset successfully." }. Token must be unused and within its 30-minute window. Proxies to UserService.

User user-service · /user

POST/user/create-userRegister a user

Auth: public · Request

UserDto{ "username": "jdoe", "email": "jdoe@x.com",
  "password": "••••••••", "userRole": "THERAPIST" }

Response: 200 { "message": "User created successfully." }, or 400 { "error": … } if the username/email already exists. BCrypt-hashes the password; generates a USR… id. userRole ∈ {CLIENT, THERAPIST, ADMIN}.

POST/user/validate-userCredential check (gateway-internal)

Auth: permitAll (intended for gateway use only) · Request: { "username", "password" }.

Response

AuthResponse{ "authenticated": true, "userId": "USR…", "username": "jdoe",
  "roles": ["THERAPIST"], "accountStatus": { "enabled": true, "locked": false },
  "failureReason": null }

On failure: authenticated=false and failureReason = INVALID_CREDENTIALS | ACCOUNT_DISABLED | ACCOUNT_LOCKED.

POST/user/forgot-usernameEmail the username

Auth: public · Request: { "email": "…" } · Response: 200 generic message ("If an account with that email exists, your username has been sent.").

Side effects: looks up the account by email and emails the username via the Gmail API; silent no-op for unknown emails (no enumeration).

POST/user/forgot-passwordIssue reset token + email

Auth: public · Request: { "email": "…" } · Response: 200 generic message.

Side effects: for a known email, stores a 30-min resetPasswordToken and sends a link via the Gmail API (EmailService); silently no-ops for unknown emails.

POST/user/reset-passwordSet new password from token

Auth: public · Request: { "token": "…", "newPassword": "…" } · Response: 200. Rejects invalid/expired tokens (IllegalArgumentException → handled by GlobalExceptionHandler). Clears the token on success.

PUT/user/update-accountUpdate email/username

Auth: JWT (username resolved from token, not body) · Request

UpdateAccountRequest{ "username": "newname", "email": "new@x.com", "currentPassword": "••••" }

Response: 200 updated UserDto. Requires the correct currentPassword; blank fields are ignored.

Auth: JWT · Query: startDate, endDate (ISO date) · Response: 200 "Successfully generated slots" (or 400 if endDate < startDate / overlaps active appointments).

Appointment appointment-service · /appointment

POST/appointment/create-appointmentBook a slot

Auth: JWT (therapistId from token) · Request

BookAppointmentRequest{ "slotId": "SLT…", "clientId": "CLT…", "modeId": "…",
  "clientName": "Jane", "customPrice": 1500.00 }   // customPrice optional — overrides the mode's default fee

Response: 200 BookAppointmentResponse — { appointmentId, paymentStatus, paymentLinkUrl, clientNotified }. The payment fields are populated only when the therapist has payments enabled (otherwise null). Errors: SlotNotAvailableException (unknown/blocked slot or invalid mode), SlotAlreadyBookedException (lost the race).

Flow: validates slot + unavailable overrides + active delivery mode (→ price, or customPrice override), atomically markSlotAsBooked (0 rows ⇒ already booked), persists the appointment, emits AppointmentCreated. After commit it best-effort creates a Razorpay payment link (never fails the booking — worst case paymentStatus=LINK_FAILED, retry via /payments/{id}/link).

PATCH/appointment/update-appointmentStatus transition

Auth: JWT · Request

UpdateAppointmentStatusRequest{ "appointmentId": "…", "status": "CONFIRMED", "reason": null }

Allowed targets: CONFIRMED | COMPLETED | CANCELLED | ABANDONED. Terminal states are immutable. CANCELLED releases the slot. Emits the matching event (AppointmentConfirmed, …). Payments: CANCELLED/ABANDONED best-effort cancel any unpaid payment link (COMPLETED is excluded so a post-session payment can still be collected). Errors: InvalidAppointmentStatusTransitionException, AppointmentNotFoundException.

PATCH/appointment/reschedule-appointmentMove to a new slot

Auth: JWT · Request

RescheduleAppointmentRequest{ "appointmentId": "…", "newSlotId": "SLT…", "modeId": "…"?, "reason": "…"? }

Response: 200 "Appointment rescheduled". Books the new slot, releases the old, re-prices by mode, sets status RESCHEDULED, emits AppointmentRescheduled (with old/new slot + times). Rejects reschedules of terminal appointments or to the same slot. Payments: since the fee may change, any unpaid link is cancelled and a fresh one issued at the new fee (a PAID payment is left untouched).

GET/appointment/get-appointmentsToday’s active appointments

Auth: JWT · Response: List<TherapistAppointments> — today’s CONFIRMED/RESCHEDULED, ordered by start time.

GET/appointment/get-availabilityEffective slots + appointments

Auth: JWT · Response: List<AvailabilityResponseDto> — effective slots joined with any booked appointment.

GET/appointment/editor-viewCalendar editor view

Auth: JWT · Query: fromDate, toDate · Response: AppointmentScheduleViewDto = { slots, appointments, overrides } for the range. 400 if toDate < fromDate.

GET/appointment/searchFiltered appointment search

Auth: JWT · Query: clientName?, status? (repeatable AppointmentStatus), fromDate, toDate · Response: List<AppointmentScheduleAppointmentDto>. Empty status ⇒ all statuses.

Payments appointment-service · /payments & /webhook

Razorpay Payment Links. Active only when the therapist has payments enabled and Razorpay keys are configured (otherwise a silent no-op — the platform's global kill switch). See the payment workflow and the Razorpay Setup Guide.

GET/payments/{appointmentId}Payment status for an appointment

Auth: JWT (scoped to therapistId) · Response: PaymentInfoDto — { appointmentId, status, amount, currency, paymentLinkUrl, razorpayPaymentId, clientNotified, failureReason, updatedAt }, or 404 if none. status ∈ {LINK_CREATED, LINK_FAILED, PAID, EXPIRED, CANCELLED}.

POST/payments/{appointmentId}/linkCreate / retry the payment link

Auth: JWT · Response: 200 PaymentInfoDto, or 422 when payments don't apply (disabled, terminal appointment, or no positive fee). Idempotent: reuses an existing LINK_CREATED/PAID link; re-issues after a LINK_FAILED/EXPIRED/CANCELLED. This is the recovery path when link creation failed during booking.

POST/webhook/razorpayRazorpay webhook (public)

Auth: none — authenticity is the X-Razorpay-Signature HMAC-SHA256 over the raw body (constant-time compared against RAZORPAY_WEBHOOK_SECRET). Invalid/missing signature → 401.

Events handled: payment_link.paid (mark PAID → auto-confirm the appointment + emit AppointmentConfirmed), payment_link.cancelled, payment_link.expired, payment.failed (records the reason; link stays active for retry). Always returns 200 after the signature check — processing is idempotent, so retries can't help and shouldn't be triggered.

Public URL: registered in the Razorpay dashboard as {host}:8091/appointment/webhook/razorpay (the gateway strips /appointment).

Admin /admin · requires ADMIN authority

The admin console authenticates via POST /auth/admin-login (8-hour token, authorities:["ADMIN"]). Every /admin/** route requires that authority at the gateway; these endpoints are spread across services.

GET/admin/servicesService health dashboard (gateway)

Response: a list of { name, status, instances } for the gateway plus the six domain services, resolved from Eureka and by pinging each service's /actuator/health (short timeouts). Status ∈ UP / DEGRADED / DOWN / UNREACHABLE.

GET/admin/usersList all users (user-service)

Response: List<AdminUserDto> (userId, username, email, role, enabled, accountLocked, failedAttempts, lastLoginTime, createdAt), newest first.

PUT/admin/users/{userId}/statusEnable / lock a user

Request: { "enabled": bool?, "locked": bool? }. Unlocking also resets the failed-attempt counter. 400 if the user isn't found.

PUT/admin/users/{userId}/reset-passwordForce-reset a password

Request: { "newPassword": "…" } (min 8 chars). BCrypt-hashes it and invalidates any pending self-service reset token.

GET/admin/auditRecent login audit (user-service)

Response: the 100 most recent UserLoginAudit rows (success + failure, IP, user-agent, reason).

GET/admin/healthService self-report (appointment & analytics)

AppointmentService returns an AdminHealthDto; AnalyticsService reports rollup freshness — { status: OK | BEHIND | NO_DATA, lastProcessedDate, daysBehind, serverTime }.

POST/admin/outbox/replayReplay outbox events (appointment-service)

Request: { "from": "YYYY-MM-DDTHH:MM:SS" } (defaults to midnight today; 400 on bad format). Re-publishes outbox events from that time — an operational recovery tool if a consumer missed messages.

Error handling

UserService, ClientService, TherapistService and AppointmentService each register a GlobalExceptionHandler (@RestControllerAdvice) mapping domain exceptions to a consistent ApiError body. Appointment-specific exceptions: SlotNotAvailableException, SlotAlreadyBookedException, AppointmentNotFoundException, InvalidAppointmentStatusTransitionException, AvailabilityProjectionConflictException.
