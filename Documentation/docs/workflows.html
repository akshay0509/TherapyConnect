Business Workflows

End-to-end traces of the core processes, showing the synchronous calls, events, projections and side effects involved in each.

1 · Login & token issuance

Validates credentials, records the attempt asynchronously, and mints a scoped JWT.

sequenceDiagram autonumber participant B as Browser participant GW as Gateway participant US as User participant TS as Therapist participant K as Kafka B->>GW: POST /auth/login GW->>US: POST /validate-user (circuit-broken) US-->>GW: AuthResponse(authenticated, roles, userId) alt not authenticated GW->>K: publish auth-login-failure GW-->>B: 401 { failureReason } else authenticated GW->>K: publish auth-login-success GW->>TS: GET /internal/therapist/user/{userId} TS-->>GW: therapistId GW->>GW: sign JWT (15m) + create refresh token GW-->>B: 200 { token } + refresh cookie end K-->>US: login event consumed → audit + lockout/reset

Async aftermath: the User service's consumers write USER_LOGIN_AUDIT, update last-login fields on success (resetting failedAttempts), and on failure increment the counter — locking the account at 5.

2 · Availability → slot generation

The therapist defines weekly rules and date overrides, then generates concrete bookable slots. Generation is the bridge between TherapistService (owner) and AppointmentService (booking engine).

sequenceDiagram autonumber participant T as Therapist (UI) participant TS as TherapistService participant DB as TS DB participant K as Kafka participant AS as AppointmentService T->>TS: POST /generate-slots?start&end TS->>TS: reject if active appointments overlap TS->>DB: delete existing slots in range TS->>DB: expand active rules × services → slots
(chop into service-duration blocks, 30-min step) TS->>DB: apply "available" overrides TS->>DB: save slots + outbox(AvailabilitySlotsGenerated) Note over TS,DB: one transaction K-->>AS: AvailabilitySlotsGenerated AS->>AS: refuse if BOOKED slots in range
else replace AVAILABLE projection rows

Slot math: for each active service, the rule's time block is chopped into slots of the service's duration, advancing in fixed 30-minute steps; a slot that would run past the block end is dropped. Each slot carries its serviceId and sessionFee. Overrides are single-day windows.

3 · Booking an appointment

The most important transaction in the system. AppointmentService validates against its local availability and delivery-mode projections, then books atomically.

sequenceDiagram autonumber participant T as Therapist (UI) participant AS as AppointmentService participant DB as AS DB participant K as Kafka participant TS as Therapist participant NS as Notification participant AN as Analytics T->>AS: POST /create-appointment {slotId, modeId, client} AS->>DB: find slot (else SlotNotAvailable) AS->>DB: check no unavailable override covers it AS->>DB: find active delivery mode → price AS->>DB: markSlotAsBooked(slotId) [affected rows] alt rows == 0 AS-->>T: SlotAlreadyBooked else booked AS->>DB: insert appointment + outbox(AppointmentCreated) AS-->>T: 200 appointmentId K-->>TS: AppointmentCreated → AppointmentProjection(SCHEDULED) K-->>AN: (ignored until Completed) end

Race safety: markSlotAsBooked is a conditional UPDATE returning the affected-row count; a return of 0 means another booking won the slot, yielding SlotAlreadyBookedException instead of a double-book. Pricing comes from the delivery mode (findByModeIdAndTherapistIdAndServiceIdAndIsActiveTrue), not the slot.

4 · Appointment state machine

Status transitions enforced by AppointmentService. Manual targets are limited to CONFIRMED/COMPLETED/CANCELLED/ABANDONED; terminal states cannot change.

stateDiagram-v2 [*] --> SCHEDULED: AppointmentCreated SCHEDULED --> CONFIRMED: confirm SCHEDULED --> RESCHEDULED: reschedule CONFIRMED --> RESCHEDULED: reschedule RESCHEDULED --> CONFIRMED: confirm CONFIRMED --> COMPLETED: complete SCHEDULED --> CANCELLED: cancel (releases slot) CONFIRMED --> CANCELLED: cancel (releases slot) SCHEDULED --> ABANDONED: abandon CONFIRMED --> ABANDONED: abandon COMPLETED --> [*] CANCELLED --> [*] ABANDONED --> [*]
  • CANCELLED releases the booked slot (markSlotAsAvailable); failing to release throws.
  • RESCHEDULED books a new slot, releases the old, re-prices by mode, and emits AppointmentRescheduled with old/new slot + times.
  • Terminal states (COMPLETED/CANCELLED/ABANDONED) reject further transitions and reschedules.

5 · Calendar sync (Notification)

Notification reacts to appointment events to manage Google Calendar invites — including a Google Meet link for online sessions.

EventAction in Google Calendar
AppointmentConfirmedCreate event with therapist+client attendees; Meet conference if ONLINE, location if OFFLINE; store mapping (idempotent — skips if mapping exists or client email unknown).
AppointmentRescheduledUpdate the existing event's times/mode (skips if no prior invite).
AppointmentCancelledDelete the event (sendUpdates=all) and remove the mapping.
CalendarBlockCreated/DeletedCreate/delete a "blocked time" calendar event when syncToGoogleCalendar is true.

6 · Analytics rollups

Analytics consumes appointment events and maintains daily aggregates + retention projections. Only certain event types affect counters:

EventEffect
AppointmentCompleted+completed; earnings += fee (or 0 if client is DSF); paid/dsf & online/offline counters; per-service rollup; bump client engagement (lastSession, totalSessions).
AppointmentCancelled+cancelled on the session's day.
AppointmentAbandoned+abandoned.
AppointmentRescheduled+rescheduled (counted against the original date).

Retention queries derive retained (>1 session), churned (no session in 30 days), retention rate, average sessions/client, and a session-frequency histogram (1, 2-3, 4-6, 7-10, 10+).

8 · Password reset

Gateway → User. User issues a 30-minute token and emails a link via the Gmail API; reset consumes the token. Always returns a generic response to avoid account enumeration. Email-send failures are logged (token stays valid for retry), not surfaced to the caller.
