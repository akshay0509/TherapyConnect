Deployment

Build tooling, the Docker Compose stack, startup ordering, and how to run the platform.

Containerized

The platform now ships a full docker-compose.yml that runs the entire backend — infrastructure, discovery, all services, and a Portainer management UI — on one host. A companion runbook (System Setup Guide) covers provisioning an Oracle Cloud VM end-to-end.

Build

A multi-module Maven build rooted at the parent POM (com.org:therapy-platform-parent:1.0.0-SNAPSHOT, packaging pom). Modules: Events, DiscoveryService, GatewayService, UserService, TherapistService, ClientService, AppointmentService, NotificationService, AnalyticsService. The React auth-app is deployed separately (Vercel).

Each service's Dockerfile is a thin JAR-copy image (FROM therapy-jdk25 · COPY target/*.jar app.jar), so Maven must produce the JARs before docker compose build.

build# 1) build the shared JDK 25 base image once (Ubuntu 24.04 + Microsoft JDK 25)
docker build -t therapy-jdk25 -f Dockerfile.base .

# 2) compile all services (Events installs first via the reactor)
mvn clean package -DskipTests

# 3) build + start the whole stack
docker compose up --build -d

Base image: every service image is FROM therapy-jdk25. Build it once from Dockerfile.base before building service images. It pulls the aarch64 JDK — on x86 hosts swap the JDK URL in Dockerfile.base.

The Compose stack

All containers join one bridge network (therapy-net) and set TZ=Asia/Kolkata. Services reach each other by container name; only the gateway (and Portainer) are published to the host.

Eureka is loopback-only. The dashboard/API has no auth and leaks the full topology, so tc-discovery publishes to 127.0.0.1:8761 — reach it via an SSH tunnel, not the public internet. Postgres and Kafka are never published.

Edge: nginx reverse proxy (host)

The gateway itself is now bound to 127.0.0.1:8091 — not internet-reachable at the Docker layer. A host-level nginx terminates TLS on 443 and reverse-proxies to 127.0.0.1:8091, setting X-Real-IP to the real client address. This does two things: it puts the whole API behind HTTPS, and it makes X-Real-IP un-spoofable (nobody can bypass nginx to forge it), which is the precondition for the gateway's per-IP login rate limiting. Keep the firewall open to only 443 + 22. Note: nginx's proxy_pass must target IPv4 127.0.0.1:8091 — an IPv6 [::1]:8091 target won't reach an IPv4-only bind.

Databases

One PostgreSQL instance hosts seven separate databases, created on first boot by postgres-init/init-databases.sql (mounted into the container's init directory): gateway, user, therapist, client, appointment, notification, analytics. Each service's DB_URL targets its own database (e.g. jdbc:postgresql://postgres:5432/appointment); tables are then created by Hibernate (ddl-auto=update). The services share one credential pair.

Keys & secrets

The ./keys directory is mounted read-only into services at /app/keys. The gateway receives both RSA keys via JWT_PRIVATE_KEY/JWT_PUBLIC_KEY (as file: references); every other service gets only the public key. All secrets come from a root .env consumed by Compose:

Eureka binding gotcha

Each service sets both EUREKA_CLIENT_SERVICEURL_DEFAULTZONE and SPRING_APPLICATION_JSON with the same defaultZone. The JSON form is what reliably binds the Eureka defaultZone inside containers — the plain env var alone did not always take, so both are set.

compose (per service)EUREKA_CLIENT_SERVICEURL_DEFAULTZONE: http://discovery:8761/eureka/
SPRING_APPLICATION_JSON: '{"eureka":{"client":{"serviceUrl":{"defaultZone":"http://discovery:8761/eureka/"}}}}'

Startup order

Compose depends_on … condition: service_healthy enforces ordering: Postgres and Kafka expose healthchecks, Discovery waits on them, and every app service waits on Postgres + Kafka + Discovery being healthy before starting.

flowchart LR A["postgres + kafka
(healthchecks)"] --> B["discovery :8761"] B --> C["gateway 127.0.0.1:8091
user · therapist · client
appointment · notification · analytics"] C --> N["host nginx :443 (TLS, X-Real-IP)"] N --> D["auth-app (Vercel)"]

Resilient to partial startup: gateway circuit breakers return SERVICE_UNAVAILABLE rather than hanging, and Kafka consumers (auto-offset-reset=earliest) replay anything emitted while they were down. restart: unless-stopped plus enabled Docker auto-start means the stack self-recovers after a reboot.

Everyday commands

operationsdocker compose up -d                 # start everything (built images)
docker compose up --build -d         # rebuild changed images + start
docker compose ps                    # container status / health
docker compose logs -f gateway       # follow one service
docker compose build appointment && docker compose up -d appointment  # ship one service
docker compose down                  # stop (keeps volumes/data)
docker compose down -v               # DESTROYS postgres_data + kafka_data

Never run docker compose down -v against a live system — -v deletes the named volumes (postgres_data, kafka_data) and all data with them.

Observability & management

  • Actuator health — each service exposes management.endpoints.web.exposure.include=health. The admin dashboard's GET /admin/services pings these to report per-service status.
  • Portainer (:9000) — container/stack management UI, mounted onto the Docker socket.
  • JSON file logging — every container caps logs (max-size/max-file) so disk can't fill; a shared ./logs volume is mounted for app logs.

Frontend (Vercel)

The React SPA in auth-app/ is built and hosted by Vercel from the repo (root directory auth-app, Vite preset). It calls only the gateway via VITE_API_BASE_URL — the nginx HTTPS endpoint (443) in front of the gateway. The refresh cookie is SameSite=None; Secure to survive the cross-site call from the Vercel origin, which requires the backend to be served over TLS (now handled by nginx).

Production hardening still open

  • Switch ddl-auto=update → validate and introduce versioned migrations (Flyway/Liquibase).
  • Move secrets from .env to a secret manager; the gateway signing key is already env-only (no longer on the classpath).
  • Raise Kafka replication above 1 and add a CI pipeline (none yet) running mvn verify.
  • TLS is now terminated by the host nginx (443) in front of the loopback-bound gateway — keep the Let's Encrypt certificate auto-renewing and the firewall limited to 443 + 22.
