Deployment

Build tooling, required infrastructure, startup ordering and how to run the platform locally.

Build

A multi-module Maven build rooted at the parent POM (com.org:therapy-platform-parent:1.0.0-SNAPSHOT, packaging pom). Modules: Events, DiscoveryService, GatewayService, UserService, TherapistService, ClientService, AppointmentService, NotificationService, AnalyticsService. The React auth-app is built separately with Vite.

build all backend modules# from repo root (Events must install first as a dependency)
mvn clean install

# or build a single service (uses the wrapper)
cd AppointmentService
./mvnw clean package        # mvnw.cmd on Windows

Build order matters: Events is a dependency of every service, so it must be installed to the local Maven repo before (or as part of) building the others. A root mvn install handles this via the reactor.

Required infrastructure

🐘 PostgreSQL

Reachable at DB_URL. Schema auto-created by Hibernate on first boot.

🔀 Apache Kafka

Brokers at KAFKA_BOOTSTRAP_SERVERS. Topics auto-created by the gateway.

🔎 Eureka

The DiscoveryService itself — start it before the others.

Startup order

Because services register with Eureka and the gateway routes via discovery, bring components up in this order:

flowchart LR A["1 · PostgreSQL + Kafka"] --> B["2 · DiscoveryService :8761"] B --> C["3 · Domain services
User, Therapist, Client,
Appointment, Notification, Analytics"] C --> D["4 · GatewayService :8091
(creates Kafka topics)"] D --> E["5 · auth-app (Vite) :5173"]
  1. Infra — PostgreSQL and Kafka must be up first.
  2. Discovery — Eureka registry on 8761.
  3. Domain services — start in any order; they register with Eureka and begin consuming Kafka. (Therapist should be available for the gateway's login therapistId lookup.)
  4. Gateway — declares topics and starts routing/auth.
  5. Frontend — npm run dev in auth-app.

The system is resilient to partial startup: circuit breakers on the gateway return SERVICE_UNAVAILABLE rather than hanging, and Kafka consumers with auto-offset-reset=earliest catch up on events emitted while they were down.

Running a service

run locally# Discovery first
cd DiscoveryService && ./mvnw spring-boot:run

# then each domain service (ensure its .env is populated)
cd UserService && ./mvnw spring-boot:run
cd TherapistService && ./mvnw spring-boot:run
# ... Client, Appointment, Notification, Analytics ...

# gateway last
cd GatewayService && ./mvnw spring-boot:run

# frontend
cd auth-app && npm install && npm run dev

Frontend build

auth-appnpm run dev       # Vite dev server on :5173
npm run build     # production bundle to dist/
npm run preview   # serve the built bundle

Configure the API base (gateway http://<host>:8091) via the app's .env (see .env.example). The SPA calls only the gateway.

Containerization & CI/CD

⚠️ Not yet present

The repository currently has no Dockerfiles, Docker Compose, Kubernetes manifests or CI pipeline. Each service is a self-contained Spring Boot fat JAR (built by spring-boot-maven-plugin) and can be containerized with a standard JRE base image. Recommended next steps: a docker-compose.yml for Postgres + Kafka + Eureka + services for local parity, then per-service images and a CI build that runs mvn verify.

Production considerations

  • Switch spring.jpa.hibernate.ddl-auto from update to validate and introduce versioned migrations (Flyway/Liquibase).
  • Externalize the RSA signing key and all secrets to a secret manager; stop shipping keys on the classpath.
  • Set Kafka topic replication > 1 and tune partitions for throughput (currently 3/1).
  • Disable show-sql and raise log levels for production.
  • Add health/readiness probes (Spring Boot Actuator is not currently wired) before orchestration.
