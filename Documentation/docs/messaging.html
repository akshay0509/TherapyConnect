Messaging & Events

Kafka is the primary integration fabric. This page documents the topics, event schemas, who produces and consumes each, and how the outbox publishes them.

Topics

Topics are auto-created by the Kafka broker (KRaft, single node); the Gateway's TopicsConfig also declares the core set at 3 partitions / 1 replica. Messages are JSON-serialized with Spring's JsonSerializer.

Type headers are OFF for outbox topics

The domain topics (client-events, therapist-availability-events, therapist-appointment-events) now run with Kafka type headers disabled end-to-end: producers set add.type.headers=false and consumers set use.type.headers=false with value.default.type=JsonNode. Every consumer deserializes to a generic JsonNode and dispatches on the payload's eventType string — so __TypeId__ is never read and trusted.packages no longer applies. Exception: the login pipeline (auth-login-*) still uses typed headers (use.type.headers=true, trusted.packages=com.org.events.login). Don't re-add typed-header/trusted-package restrictions to the outbox topics.

Event schemas

All event classes live in the shared com.org.events library. The eventType string field is the discriminator consumers switch on.

AppointmentEvent — topic therapist-appointment-events

Event types: AppointmentCreated, AppointmentConfirmed, AppointmentCompleted, AppointmentCancelled, AppointmentAbandoned, AppointmentRescheduled.

fieldseventId, eventType, appointmentId, slotId, serviceId, modeId,
modeType, address, sessionFee, therapistId, clientId,
startTime, endTime, bookingSource,
oldSlotId, newSlotId, oldStartTime, oldEndTime,   // reschedule
updatedAt, cancelledBy, reason

Consumed by: Therapist (builds AppointmentProjection), Notification (Google Calendar invite on Confirmed/Rescheduled/Cancelled), Analytics (rollups on Completed/Cancelled/Abandoned/Rescheduled).

AvailabilityEvent & batch events — topic therapist-availability-events

Single-slot AvailabilityEvent types: AvailabilitySlotCreated, AvailabilitySlotRemoved (fields: slotId, therapistId, serviceId, sessionFee, start/end, generationId). Batch variants: AvailabilitySlotsGeneratedEvent (range + list of Slot) and AvailabilitySlotsDeletedEvent (range). Also on this topic: CalendarBlockEvent (CalendarBlockCreated/Deleted, with syncToGoogleCalendar) and DeliveryModeEvent.

Consumed by: Appointment (materializes/clears its slot projection; refuses to wipe BOOKED slots), Notification (Google Calendar block events).

ClientEvent — topic client-events

Types: ClientCreated, ClientUpdated, ClientStatusUpdated. Fields: eventId, eventType, occurredAt, clientId, therapistId, email, phoneNumber, firstName, lastName, status (ACTIVE/TERMINATED), dsf.

Consumed by: Therapist (updates THERAPIST_CLIENTS name/status), Notification (ClientProjection email — now upserts on ClientUpdated too; previously only ClientCreated was consumed and contact edits never reached the invite email), Analytics (DSF projection), and Appointment (ClientContactProjection email/phone for Razorpay notify).

TherapistEvent — topic therapist-availability-events

Types: TherapistCreated, TherapistUpdated, TherapistPaymentSettingsUpdated. Fields: eventId, eventType, occurredAt, therapistId, timezone, email, paymentEnabled. TherapistUpdated fires when the therapist email is changed from Account Settings so downstream projections follow.

Consumed by: Appointment's TherapistPaymentSettingsConsumer (group appointment-therapist-payment-projection-group) — upserts THERAPIST_PAYMENT_PROJECTION so booking knows whether to create Razorpay links — and Notification, which upserts THERAPIST_PROJECTION (timezone + email) so calendar invites use the therapist's own email rather than a single env-configured address.

LoginSuccessEvent / LoginFailureEvent — topics auth-login-*

Carry userId, username, ipAddress, userAgent, timestamp (failure adds reason). Consumed by: User — writes USER_LOGIN_AUDIT, updates last-login fields, and increments/locks on failures (5 → locked) or resets the counter on success.

These are the only events published without the outbox — the gateway sends them directly via KafkaTemplate for low latency on the auth path.

Outbox publication loop

For domain events (client/availability/appointment), production goes through the outbox so the event and the state change commit together. The publisher is a per-service scheduled job:

OutboxEventScheduler@Scheduled(fixedDelay = 2000)
@Transactional
public void publishPendingEvents() {
    var events = repo.findTop100ByPublishedFalseOrderByCreatedAtAsc();
    for (OutboxEvent e : events) {
        try {
            producer.sendMessage(e.getAggregateId(), e.getPayload()); // blocking .get()
            e.setPublished(true);
            repo.save(e);
        } catch (Exception ex) {
            log.error("Failed to publish {}", e.getOutboxEventId(), ex);
            break;   // stop the batch, preserve ordering, retry next tick
        }
    }
}

The producer publishes the raw jsonb payload keyed by aggregateId, so all events for one therapist/appointment land on the same partition and stay ordered. The Kafka send is awaited (.get()) so a failure is caught and the row stays unpublished.

Poison-event parking

A permanently unpublishable event (oversized record, serialization failure) used to block everything behind it forever, since the scheduler stops the batch on failure to preserve ordering. The outbox now distinguishes the two failure classes:

  • Transient (broker down/slow) — increment retryCount, stop the batch, retry next tick. Ordering preserved.
  • Fatal (RecordTooLargeException, SerializationException) or 150 retries (~5 minutes of 2s ticks — long enough that a broker restart never parks a healthy event) — mark the row parked=true and continue past it.

Parked events are excluded from the scheduler's query and from the admin health "pending" count (they'd keep the card red forever); they get their own parkedCount. The only way back into rotation is an explicit admin outbox replay, which un-parks rows in the requested range with a fresh retry budget — deliberate, because replaying a stale parked event after newer events for the same entity can regress projections.

Retry & dead-letter pipeline

Every consuming service (Appointment, Therapist, Notification, Analytics) registers a KafkaConsumerConfig with a Spring Kafka DefaultErrorHandler:

  1. A listener exception triggers 2 retries with a 5s fixed backoff (FixedBackOff(5000, 2)).
  2. If it still fails, a DeadLetterPublishingRecoverer publishes the record to <topic>.DLT (partition -1 — producer-chosen, since auto-created DLTs have fewer partitions than the 3-partition source topics). Kafka's standard kafka_dlt-original-topic / kafka_dlt-exception-message headers travel with it.
  3. Consumers are therefore written to let failures propagate rather than catch-and-log. Notification is the poster child: it previously swallowed Google Calendar errors (expired-token failures silently lost events for days); it now throws, retries, and dead-letters so the failure is visible and replayable.

Monitoring & replay: the gateway's KafkaMonitorService powers the admin dashboard (/admin/kafka): DLT depth, consumer-group lag, message peek, and one-click replay that republishes DLT messages to their original topic. "Pending" is measured against the committed offsets of the dedicated dlt-replay consumer group — topics are append-only, so the committed offset is what separates handled from unhandled. Replay is safe because every consumer is idempotent.

Delivery semantics

Delivery is at-least-once end to end: a crash after Kafka ack but before the outbox's published=true commit republishes, and a DLT replay can redeliver an already-applied event. Consumers tolerate both via existence checks before applying.

Idempotency patterns in consumers

  • Appointment availability consumer — existsBySlotId before insert; never deletes a BOOKED slot (throws AvailabilityProjectionConflictException on a destructive sync that would hit booked slots).
  • Notification — checks for an existing AppointmentCalendarEvent mapping before creating a Google event; skips reschedule if no prior invite.
  • Therapist appointment consumer — upserts the projection by appointmentId; warns and no-ops if a status/reschedule event arrives for an unknown appointment.
  • Razorpay webhook — not Kafka, but idempotent all the same: a PAID payment row is never transitioned again, so duplicate payment_link.paid deliveries are safe (Razorpay retries until it gets a 200).
