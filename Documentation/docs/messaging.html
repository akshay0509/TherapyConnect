Messaging & Events

Kafka is the primary integration fabric. This page documents the topics, event schemas, who produces and consumes each, and how the outbox publishes them.

Topics

All topics are declared by the Gateway's TopicsConfig as 3 partitions / 1 replica (dev settings). Messages are JSON-serialized with Spring's JsonSerializer; type headers are added by producers, and consumers either trust the headers or deserialize generically into a JsonNode and branch on an eventType field.

Event schemas

All event classes live in the shared com.org.events library. The eventType string field is the discriminator consumers switch on.

AppointmentEvent — topic therapist-appointment-events

Event types: AppointmentCreated, AppointmentConfirmed, AppointmentCompleted, AppointmentCancelled, AppointmentAbandoned, AppointmentRescheduled.

fieldseventId, eventType, appointmentId, slotId, serviceId, modeId,
modeType, address, sessionFee, therapistId, clientId,
startTime, endTime, bookingSource,
oldSlotId, newSlotId, oldStartTime, oldEndTime,   // reschedule
updatedAt, cancelledBy, reason

Consumed by: Therapist (builds AppointmentProjection), Notification (Google Calendar invite on Confirmed/Rescheduled/Cancelled), Analytics (rollups on Completed/Cancelled/Abandoned/Rescheduled).

AvailabilityEvent & batch events — topic therapist-availability-events

Single-slot AvailabilityEvent types: AvailabilitySlotCreated, AvailabilitySlotRemoved (fields: slotId, therapistId, serviceId, sessionFee, start/end, generationId). Batch variants: AvailabilitySlotsGeneratedEvent (range + list of Slot) and AvailabilitySlotsDeletedEvent (range). Also on this topic: CalendarBlockEvent (CalendarBlockCreated/Deleted, with syncToGoogleCalendar) and DeliveryModeEvent.

Consumed by: Appointment (materializes/clears its slot projection; refuses to wipe BOOKED slots), Notification (Google Calendar block events).

ClientEvent — topic client-events

Types: ClientCreated, ClientUpdated, ClientStatusUpdated. Fields: eventId, eventType, occurredAt, clientId, therapistId, email, firstName, lastName, status (ACTIVE/TERMINATED), dsf.

Consumed by: Therapist (updates THERAPIST_CLIENTS name/status), Notification (ClientProjection email), Analytics (DSF projection).

LoginSuccessEvent / LoginFailureEvent — topics auth-login-*

Carry userId, username, ipAddress, userAgent, timestamp (failure adds reason). Consumed by: User — writes USER_LOGIN_AUDIT, updates last-login fields, and increments/locks on failures (5 → locked) or resets the counter on success.

These are the only events published without the outbox — the gateway sends them directly via KafkaTemplate for low latency on the auth path.

Outbox publication loop

For domain events (client/availability/appointment), production goes through the outbox so the event and the state change commit together. The publisher is a per-service scheduled job:

OutboxEventScheduler@Scheduled(fixedDelay = 2000)
@Transactional
public void publishPendingEvents() {
    var events = repo.findTop100ByPublishedFalseOrderByCreatedAtAsc();
    for (OutboxEvent e : events) {
        try {
            producer.sendMessage(e.getAggregateId(), e.getPayload()); // blocking .get()
            e.setPublished(true);
            repo.save(e);
        } catch (Exception ex) {
            log.error("Failed to publish {}", e.getOutboxEventId(), ex);
            break;   // stop the batch, preserve ordering, retry next tick
        }
    }
}

The producer publishes the raw jsonb payload keyed by aggregateId, so all events for one therapist/appointment land on the same partition and stay ordered. The Kafka send is awaited (.get()) so a failure is caught and the row stays unpublished.

⚠️ Delivery semantics & DLQ

Delivery is at-least-once: a crash after Kafka ack but before the published=true commit will republish. Consumers are written to be idempotent (existence checks before applying). A dedicated dead-letter topic/DLT is referenced in Analytics' Kafka config but a full DLQ + retry policy is not yet implemented platform-wide — see Technical Debt.

Idempotency patterns in consumers

  • Appointment availability consumer — existsBySlotId before insert; never deletes a BOOKED slot (throws AvailabilityProjectionConflictException on a destructive sync that would hit booked slots).
  • Notification — checks for an existing AppointmentCalendarEvent mapping before creating a Google event; skips reschedule if no prior invite.
  • Therapist appointment consumer — upserts the projection by appointmentId; warns and no-ops if a status/reschedule event arrives for an unknown appointment.
