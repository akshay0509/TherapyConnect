Security

Authentication, authorization, token lifecycle, abuse protection and data-at-rest encryption.

Trust model

The system uses asymmetric JWT. The Gateway is the sole token issuer and holds the RSA private key; every backend service is a stateless OAuth2 resource server that holds only the matching public key and verifies signatures locally. No service can mint tokens; no shared session store exists.

Every backend builds a JwtDecoder from the public key (NimbusJwtDecoder.withPublicKey(...)) and enables oauth2ResourceServer().jwt(). Controllers never trust client-supplied identity — they call SecurityUtils.getTherapistId() which reads the verified JWT from the security context. This is the core multi-tenant isolation mechanism.

JWT propagation between services

When TherapistService calls ClientService, JwtPropagationInterceptor re-attaches the caller's bearer token so the downstream resource server can authenticate the same principal. There is no separate service-account credential.

Refresh tokens

Opaque UUID tokens stored in REFRESH_TOKENS (not JWTs), delivered as a hardened cookie and rotated on every use.

  • Cookie: httpOnly, secure, SameSite=None, path scoped to /auth/refresh, 7-day max-age. (SameSite=None so the cookie survives the cross-site call from the Vercel-hosted SPA to the backend — it requires Secure.)
  • Rotation: /auth/refresh deletes the presented token and issues a new one (creating a token deletes any prior tokens for the username — single active session per user).
  • Roles preserved: the row now stores the user's roles (and back-fills therapistId if it was null at first login), so refreshed JWTs carry full authorities.
  • Revocation: /auth/logout deletes the user's tokens and expires the cookie.
  • Expiry check: isExpiredOrRevoked rejects revoked or past-expiryDate tokens.

Fixed: earlier builds minted refresh tokens with an empty authorities claim because roles weren't stored. The roles column resolves that — role-based authorization (e.g. the ADMIN authority) now works across refreshes.

Admin authentication

The admin console is a separate, credential-based path — not tied to the user table.

  • Login: POST /auth/admin-login checks the submitted username/password against the ADMIN_USERNAME/ADMIN_PASSWORD env values using a constant-time comparison (MessageDigest.isEqual) to avoid timing leaks.
  • Token: an 8-hour RS256 JWT with authorities:["ADMIN"] and userId:"admin". No refresh token — admins re-login when it expires.
  • Authorization: the gateway's JwtAuthenticationConverter reads the authorities claim verbatim (no ROLE_ prefix); /admin/** is gated by hasAuthority("ADMIN").
  • Rate limit: adminLoginRateLimiter — 5/min (stricter than the 10/min user login, since admin has no lockout).

The admin account is a single static credential pair from the environment. Rotate it, keep it out of VCS, and consider promoting to per-admin accounts + MFA before exposing the console beyond a trusted network.

The gateway explicitly denies /*/internal/** so service-internal endpoints (e.g. /therapist/internal/...) can never be reached from the public edge — the gateway's own proxy calls them directly over the Docker network. CSRF is disabled across services (stateless API; tokens are bearer/JWT, refresh cookie is SameSite=None; Secure). CORS is configured at the gateway (CorsConfig) for the SPA origin.

Abuse & failure protection

🚦 Login rate limiting

Resilience4j @RateLimiter("loginRateLimiter"): 10 requests/min, 0s wait → fast 429 via loginRateLimitFallback.

🔒 Account lockout

5 failed attempts (counted by the login-failure consumer) sets isAccountLocked=true; a success resets the counter to 0.

⚡ Circuit breakers

userService & therapistService breakers (50% failure over 10 calls, 30s open). On open, login fails closed with SERVICE_UNAVAILABLE.

🕵️ Login auditing

Every attempt (success/failure, IP, UA, reason) recorded in USER_LOGIN_AUDIT.

Account-enumeration resistance

Forgot-password always returns a generic "if the email exists…" message, and the User service silently no-ops for unknown emails. Email-send failures are logged but not surfaced. Login failures return a coarse failureReason enum rather than field-specific errors.

Encryption at rest

Clinical session notes are encrypted at the field level before persistence:

EncryptionServiceAES/GCM/NoPadding
IV       : 12 random bytes (SecureRandom), prepended to ciphertext
Tag      : 128-bit
Key      : Base64-decoded from ${app.encryption.secret}
Storage  : Base64(IV || ciphertext) in SESSION_NOTES.noteContent (TEXT)
Wiring   : JPA @Convert(SessionNotesEncryptor) — transparent on read/write

This protects notes if the database is dumped, provided the key (an env secret) is not also compromised.

Payment webhook verification

The Razorpay webhook (POST /webhook/razorpay) is unauthenticated by JWT — authenticity comes from the X-Razorpay-Signature header, an HMAC-SHA256 of the raw request body keyed by RAZORPAY_WEBHOOK_SECRET, hex-encoded and compared in constant time. A missing/invalid signature is rejected with 401; only after it passes is the body processed.

Resolved since the first review

The signing key is no longer on the classpath (env/volume only); refreshed tokens keep their roles; internal endpoints are denied at the gateway; and the outbox topics no longer rely on trusted.packages=* (type headers are off, nothing is resolved from headers).

Hardening still open

Restrict /user/validate-user to gateway-only network paths; give each service its own DB credentials; move the static admin credential to per-admin accounts + MFA; consider short-lived service tokens instead of propagating the user JWT; front the gateway with HTTPS so payments and the SameSite=None cookie run over TLS.
