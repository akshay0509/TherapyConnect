Kafka serialization config

Producers use StringSerializer keys + JsonSerializer values with type headers enabled. Consumers wrap delegates in ErrorHandlingDeserializer and set auto-offset-reset=earliest. Trusted packages are typically * (User narrows to com.org.events.login).

spring.json.trusted.packages=* trusts all types during deserialization — fine for a closed internal bus, but should be narrowed before exposure. Analytics also sets spring.jackson.serialization.write-dates-as-timestamps=false for ISO date output.

Profiles, feature flags & logging

  • Profiles: none are defined — configuration is single-environment driven entirely by env vars. Introducing application-prod.properties is a recommended next step.
  • Feature flags: no flag framework; behavior toggles are code-level (e.g. syncToGoogleCalendar on a calendar-block event, isActive on services/modes).
  • Logging: Log4j2 via classpath:log4j2.xml, root at ERROR, each app's own package at INFO. TherapistService adds MDC trace IDs (MdcFilter + TraceIdInterceptor).

Startup process

Each app is a standard Spring Boot main class (@SpringBootApplication). On boot a service: registers with Eureka, runs Hibernate DDL update, connects to Kafka (starting listeners), and starts its outbox scheduler if present. The Gateway additionally creates Kafka topics via TopicsConfig and loads the RSA key pair. See Deployment for ordering.
