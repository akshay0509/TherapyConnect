Configuration

Environment variables, application properties, key infrastructure settings and the startup process.

Environment variables

Each backend service reads secrets/infra from the environment. In the Docker Compose deployment these come from a single root .env and are injected per service (with the per-service DB_URL, JWT key paths, Eureka zone, and TZ=Asia/Kolkata set in the compose file). Properties reference them with ${VAR} placeholders.

Kafka serialization config

Producers use StringSerializer keys + JsonSerializer values; consumers wrap delegates in ErrorHandlingDeserializer and set auto-offset-reset=earliest. For the outbox/domain topics, type headers are off:

domain-topic services (appointment/therapist/client/notification/analytics)# producers
spring.kafka.producer.properties.spring.json.add.type.headers=false
# consumers — deserialize to JsonNode and dispatch on the eventType field
spring.kafka.consumer.properties.spring.json.use.type.headers=false
spring.kafka.consumer.properties.spring.json.value.default.type=com.fasterxml.jackson.databind.JsonNode
spring.kafka.consumer.properties.reconnect.backoff.max.ms=10000

Because nothing is resolved from headers, trusted.packages no longer applies to these topics. The login pipeline is the exception: UserService keeps use.type.headers=true and trusted.packages=com.org.events.login. Analytics sets spring.jackson.serialization.write-dates-as-timestamps=false for ISO dates. Actuator health is exposed everywhere via management.endpoints.web.exposure.include=health.

# Login rate limiting — custom per-client-IP fixed window (LoginRateLimiter, Caffeine)
security.login.rate-limit.per-ip=10

security.login.rate-limit.window-seconds=60

# Circuit breakers (userService, therapistService) — still Resilience4j
failureRateThreshold=50  slidingWindowSize=10  waitDurationInOpenState=30s

The login/admin-login throttle moved off the Resilience4j global @RateLimiter (which one IP could exhaust to lock everyone out) to a per-IP limiter keyed on the nginx X-Real-IP. Resilience4j is still used for the inter-service circuit breakers.

Profiles, feature flags & logging

  • Profiles: none are defined — configuration is single-environment driven entirely by env vars. Introducing application-prod.properties is a recommended next step.
  • Feature flags: no flag framework; behavior toggles are code-level (e.g. syncToGoogleCalendar on a calendar-block event, isActive on services/modes).
  • Logging: Log4j2 via classpath:log4j2.xml, root at ERROR, each app's own package at INFO. TherapistService adds MDC trace IDs (MdcFilter + TraceIdInterceptor).

Startup process

Each app is a standard Spring Boot main class (@SpringBootApplication). On boot a service: registers with Eureka, runs Hibernate DDL update, connects to Kafka (starting listeners), and starts its outbox scheduler if present. The Gateway additionally creates Kafka topics via TopicsConfig and loads the RSA key pair. See Deployment for ordering.
