Onboarding

A guided path for a new engineer: get it running, learn the conventions, and make a safe first change.

Build the mental model first

Before touching code, internalize three things from the Architecture page:

  1. The gateway is the only door. The SPA never calls a backend service directly. Auth happens once, at the gateway, and produces a JWT carrying therapistId.
  2. Services own their data and talk via Kafka. If service A needs B's data, it keeps a projection updated from B's events — it doesn't query B's tables.
  3. Writes use the outbox. A domain change and its event commit together; a 2-second scheduler publishes them.

  1. Start PostgreSQL and Kafka.
  2. Populate each service's .env (DB_URL, DB_USERNAME, DB_PASSWORD, KAFKA_BOOTSTRAP_SERVERS, plus APP_ENCRYPTION_SECRET for Therapist and Google vars for Notification/User). Generate an RSA key pair into each service's resources/keys/ (private only in the gateway).
  3. mvn clean install at the root (installs Events first).
  4. Run in order: Discovery → domain services → Gateway → frontend. The fastest path is the Docker Compose stack — build the therapy-jdk25 base image, mvn clean package, then docker compose up --build -d (see Deployment).
  5. Create a user via POST /user/create-user, then log in at POST /auth/login. For the admin console, log in at POST /auth/admin-login with the ADMIN_* env credentials.

Need an AES key for notes? Any Base64-encoded 16/24/32-byte value works for APP_ENCRYPTION_SECRET in dev. Lose it and existing encrypted notes become unreadable.

Code conventions

  • Package layout is identical across services: Controller, Services, Repository, Entity, Dto, Enums, Messaging, Config, Utility (PascalCase package names).
  • Identity always comes from the JWT via SecurityUtils.getTherapistId()/getUserId() — never from request bodies.
  • Lombok @Data on entities/DTOs/events; don't hand-write getters/setters.
  • IDs are prefixed UUID strings generated in @PrePersist (THP, CLT, SLT, …).
  • Events live only in the shared com.org.events module and branch on a string eventType.
  • Writes that others must hear about go through OutboxService.saveOutboxEvent(...) inside the same @Transactional method.

A safe first change (worked example)

"Add a notes field to a client." Because client data crosses services via events, this touches more than one place — a perfect lesson in the architecture:

  1. Add the column to Client (client-service) and the ClientDto + assembler.
  2. If other services need it, add it to ClientEvent in the Events module and set it in ClientService.publishClientEvent.
  3. Update consumers that should store it (e.g. Therapist's THERAPIST_CLIENTS projection) — and keep them idempotent.
  4. Expose it through the relevant controller. Test the full path: create client → observe the event → confirm the projection updated.

Golden rule: never reach into another service's tables to "just read" a value. Add it to the event and project it. That's what keeps the services independently deployable.

Debugging tips

  • Event not arriving? Check the producer's OUTBOX_EVENT table — is the row published=true? If not, the scheduler or Kafka is the problem.
  • Consumer stuck? ErrorHandlingDeserializer isolates poison messages; check logs for deserialization errors. auto-offset-reset=earliest means a fresh group replays everything.
  • 403/401 on a backend call? The JWT may lack the expected claim (e.g. therapistId absent → SecurityUtils throws). Note /*/internal/** is denied at the gateway, and /admin/** needs the ADMIN authority.
  • Trace a request in therapist-service via the MDC trace id added by MdcFilter.
  • Booking conflicts? Remember markSlotAsBooked/markSlotAsAvailable are conditional updates — a 0-row result is the signal a race or terminal state was hit.
