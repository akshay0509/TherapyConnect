Architecture

How the services fit together, the patterns that hold them together, and how data flows through the system.

Architectural style

TherapyConnect is a domain-partitioned microservices system with event-driven choreography. There is no central orchestrator: services collaborate by publishing and consuming Kafka events, and each service owns its data. Synchronous REST is used only at the edges — the gateway calling the User and Therapist services during login, and the Therapist service calling the Client service to create a client.

Three patterns recur throughout and are worth internalizing before reading any single service:

🗃️ Transactional Outbox

State change and the event describing it are written in one DB transaction to an OUTBOX_EVENT table. A scheduler polls and publishes to Kafka, guaranteeing at-least-once delivery without dual-write loss.

📐 CQRS read-model projections

Services keep local read-only copies (projections) of data they don't own, built from events — e.g. AppointmentService projects availability slots; TherapistService projects appointments.

🔑 Stateless JWT trust

The gateway is the single token issuer (RS256). Every downstream service independently verifies the signature with a shared public key — no session store, no per-request auth callback.

System topology

The browser only ever talks to the Gateway (port 8091). The gateway authenticates, then routes to backend services discovered through Eureka (port 8761). Backend services talk to PostgreSQL and exchange events through Kafka.

The transactional outbox in detail

This is the backbone of reliable eventing. AppointmentService, ClientService and TherapistService all implement it identically:

  1. A @Transactional service method mutates domain tables (e.g. marks a slot booked, inserts an appointment).
  2. In the same transaction it calls OutboxService.saveOutboxEvent(...), which serializes the event to a jsonb payload and inserts a row into OUTBOX_EVENT with published=false.
  3. A separate OutboxEventScheduler runs every 2 seconds (@Scheduled(fixedDelay = 2000)), selects up to 100 unpublished rows oldest-first, and publishes each to Kafka.
  4. On successful publish the row is flipped to published=true. On failure the loop breaks, preserving ordering and leaving the row for the next tick (at-least-once).

⚠️ Consumer idempotency matters

Because delivery is at-least-once, every consumer must tolerate duplicates. The codebase does this with existence checks (existsBySlotId, findById(...).isPresent()) before applying an event. The Gateway's login events bypass the outbox and publish directly — a deliberate trade-off for the auth hot path.

Read-model projections (CQRS)

No service queries another service's database. Instead, each service materializes exactly the slice of foreign data it needs, kept current by Kafka consumers. This decouples runtime availability (a down service doesn't block reads) at the cost of eventual consistency.

Shared events library

The Events Maven module (com.org:events) is a dependency of every service and contains only the event POJOs and their enums (AppointmentEvent, ClientEvent, AvailabilityEvent, LoginSuccessEvent, status enums, …). This gives producers and consumers a single shared contract and is the one piece of code intentionally coupled across the whole system. Changing a class here ripples to every service — treat it as a versioned API.

Database isolation — partial

Each service now connects to its own physical database within a single PostgreSQL instance (created by postgres-init), so tables are no longer co-mingled. However all services still share one credential pair (DB_USERNAME/DB_PASSWORD), so the isolation is by database, not by least-privilege user. Per-service credentials remain a hardening item — see Technical Debt.

Beyond the therapist workflow

Two capabilities sit alongside the core domain: an admin console (separate credential-based login at the gateway, ADMIN-gated /admin/** endpoints for user management, login audit, service health and outbox replay) and Razorpay payments (AppointmentService owns the payment-link lifecycle; an HMAC-verified webhook auto-confirms paid appointments). Both are optional and fail closed when unconfigured. See Workflows and Security.
