Oracle Cloud Free Tier — Account & VM Provisioning

Oracle Cloud Free Tier gives you an Always Free VM that can comfortably run this entire stack. The ARM-based (Ampere) instances are strongly preferred — you get up to 4 OCPUs and 24 GB RAM on the free tier, which is far more than the x86 free instances.

1.1 Create an Oracle Cloud Account

  1. Go to cloud.oracle.com and click Start for free.
  2. Fill in name, country, and email. Verify the email address.
  3. Enter a credit card (required for verification, not charged for Always Free resources).
  4. Choose your home region carefully — you cannot change it later. Pick the region geographically closest to your users (e.g. ap-mumbai-1 for India).
  5. Complete provisioning. The account may take 5–15 minutes to activate.

1.2 Create the VM Instance

  1. From the OCI Console, open the hamburger menu → Compute → Instances → Create Instance.
  2. Name: give it a descriptive name, e.g. therapy-vm.
  3. Image: click Change image → select Canonical Ubuntu, version 22.04. Do not pick Oracle Linux unless you specifically want it.
  4. Shape: click Change shape → select Ampere (ARM) → choose VM.Standard.A1.Flex. Set OCPUs to 2 and Memory to 12 GB (well within the free tier limit of 4 OCPUs / 24 GB total across all ARM instances).
  5. Networking: leave the defaults (a new VCN will be created automatically, or pick an existing one). Ensure Assign a public IPv4 address is checked — you need a public IP for DuckDNS.
  6. SSH Keys: click Save Private Key to download the auto-generated .key file, or paste in your own public key. Keep this key file safe — it's the only way in.
  7. Boot Volume: the default 50 GB is sufficient. You can increase it up to 200 GB total across Always Free instances.
  8. Click Create. The instance will take 1–2 minutes to reach the Running state.
  9. Note the Public IP address shown on the instance details page — you'll need it in multiple places.
Always Free limits: 4 ARM OCPUs and 24 GB RAM total across all Ampere instances. You can run one instance with all 4 OCPUs / 24 GB, or split across multiple instances, as long as the totals stay within the limit.

Oracle VCN Security List — Open Ports in OCI Console

Oracle Cloud wraps every VM in a Virtual Cloud Network (VCN) with a Security List that acts like a cloud-level firewall. By default, only port 22 (SSH) is open. You must explicitly add ingress rules for every port you want accessible from the internet.

Two-layer firewall: Oracle has BOTH a cloud security list (configured here in OCI Console) AND an OS-level iptables firewall inside Ubuntu. You must open ports in BOTH places. See Section 4 for iptables.

Steps to Add Ingress Rules

  1. In the OCI Console, go to Networking → Virtual Cloud Networks.
  2. Click the VCN that was created with your instance (usually named something like vcn-YYYYMMDD-xxxx).
  3. In the left panel, click Security Lists → click the Default Security List.
  4. Click Add Ingress Rules.
  5. Add the following rules one at a time:
    Source CIDRProtocolPortPurpose
    0.0.0.0/0TCP8091API Gateway (all frontend traffic)
    0.0.0.0/0TCP8761Eureka dashboard (optional — remove if not needed externally)
  6. For each rule: Source Type = CIDR, Source CIDR = 0.0.0.0/0, IP Protocol = TCP, enter the port number. Click Add Ingress Rules.
Note: You do not need to open PostgreSQL (5432) or Kafka (9092) — those ports are never mapped to the host in docker-compose.yml and should remain completely internal.

Connect via SSH

The default username for Ubuntu images on Oracle Cloud is ubuntu.

From your local machine (Linux / macOS / Git Bash / WSL)

chmod 400 ~/Downloads/ssh-key-YYYY-MM-DD.key

From Windows (PowerShell or Command Prompt)
ssh -i C:\Users\YourName\Downloads\ssh-key-YYYY-MM-DD.key ubuntu@YOUR_PUBLIC_IP

Update the System

Always do a full system update on first login.

sudo apt-get update && sudo apt-get upgrade -y

Set Up a Swap File (Recommended)

Even with 12 GB RAM, a swap file prevents OOM kills during Maven builds, which are memory-intensive.

# Create a 4 GB swap file
sudo fallocate -l 4G /swapfile
sudo chmod 600 /swapfile
sudo mkswap /swapfile
sudo swapon /swapfile

# Make it persist across reboots
echo '/swapfile none swap sw 0 0' | sudo tee -a /etc/fstab

# Verify swap is active
free -h

Add ubuntu User to the Docker Group (do this after installing Docker)

After Docker is installed (Section 5), run this so you never need sudo before docker commands:

sudo usermod -aG docker ubuntu
# Log out and back in for the group change to take effect
exit

Ubuntu Firewall (iptables) — Open Ports on the OS Level

Oracle's Ubuntu 22.04 image ships with an iptables rule that drops all incoming traffic except SSH, even after you open the port in the OCI security list. This is the most common source of confusion when setting up the system — the cloud firewall is open but the OS drops the packet anyway.

Critical: if you skip this step, port 8091 will time out from the browser even though the OCI security list is correctly configured. Both layers must be open.

Option A — iptables (direct, recommended for Oracle Ubuntu)

# Check what rules currently exist
sudo iptables -L INPUT --line-numbers -n

# Open port 8091 (API Gateway)
sudo iptables -I INPUT 6 -m state --state NEW -p tcp --dport 8091 -j ACCEPT

# Open port 8761 (Eureka, optional)
sudo iptables -I INPUT 6 -m state --state NEW -p tcp --dport 8761 -j ACCEPT

# Save the rules so they survive reboots
sudo apt-get install -y iptables-persistent
sudo netfilter-persistent save

# Verify rules were saved
sudo iptables -L INPUT --line-numbers -n
Why -I INPUT 6? Oracle's default iptables chain has a REJECT rule at a specific position. Inserting at position 6 (or using -I INPUT to prepend) ensures the ACCEPT rule fires before the REJECT rule. Using -A (append) would add it after the REJECT and have no effect.

Option B — UFW (simpler but requires disabling Oracle's chain)

# Only use this if you prefer UFW over raw iptables
sudo ufw allow 22
sudo ufw allow 8091
sudo ufw allow 8761
sudo ufw enable
sudo ufw status

Verify from the Outside

After opening ports, do a quick test from your local machine (after Docker services are running):

# Should return the Eureka HTML page
curl http://YOUR_PUBLIC_IP:8761/

# Should return a gateway response (likely 401 Unauthorized — that's correct)
curl http://YOUR_PUBLIC_IP:8091/actuator/health

Install Docker & Docker Compose

All services run inside Docker. Docker Compose (v2, the plugin version) is used to manage the whole stack. Do not install the old standalone docker-compose — use the plugin (docker compose).

Install Docker Engine (Official Method)

# Remove any old Docker packages that might conflict
for pkg in docker.io docker-doc docker-compose docker-compose-v2 podman-docker containerd runc; do
  sudo apt-get remove -y $pkg 2>/dev/null; done

# Install prerequisites
sudo apt-get update
sudo apt-get install -y ca-certificates curl

# Add Docker's official GPG key
sudo install -m 0755 -d /etc/apt/keyrings
sudo curl -fsSL https://download.docker.com/linux/ubuntu/gpg \
  -o /etc/apt/keyrings/docker.asc
sudo chmod a+r /etc/apt/keyrings/docker.asc

# Add the Docker apt repository
echo \
  "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.asc] \
  https://download.docker.com/linux/ubuntu \
  $(. /etc/os-release && echo "$VERSION_CODENAME") stable" | \
  sudo tee /etc/apt/sources.list.d/docker.list > /dev/null

# Install Docker Engine + Compose plugin
sudo apt-get update
sudo apt-get install -y docker-ce docker-ce-cli containerd.io \
  docker-buildx-plugin docker-compose-plugin

Post-Install: Run Docker Without sudo

sudo usermod -aG docker ubuntu
# Log out and back in, then verify:
docker --version
docker compose version

Enable Docker to Auto-Start on Reboot

sudo systemctl enable docker
sudo systemctl enable containerd

Verify Installation

docker run --rm hello-world

You should see a "Hello from Docker!" message.

Install Java 25 (JDK) & Maven — For Building Services

The Dockerfiles use a pre-built JAR file (COPY target/*.jar app.jar), which means Maven must compile the source code and produce the JAR before the Docker image is built. Java 25 and Maven must be installed on the Oracle VM itself.

Why on the VM? The build-on-VM approach keeps things simple: pull latest code → run mvn → run docker compose build. No need to push images to a registry or SCP files from a Windows machine.

Install Java 25 Using SDKMAN (Recommended)

SDKMAN is a version manager for JDK/Maven/Gradle. It's the easiest way to install non-LTS Java versions like JDK 25.

# Install SDKMAN
curl -s "https://get.sdkman.io" | bash

# Activate SDKMAN in the current shell
source "$HOME/.sdkman/bin/sdkman-init.sh"

# List available JDK 25 builds
sdk list java | grep 25

# Install the Oracle OpenJDK 25 build (use the identifier shown in the list above)
# The identifier looks like: 25-open, 25.0.1-oracle, etc.
sdk install java 25-open

# Set it as the default
sdk default java 25-open

# Verify
java -version
Note: Run sdk list java | grep "25" to see what's currently available. The identifier changes as new builds are released. Pick the one labelled open (OpenJDK) or oracle for the latest JDK 25 release.

Make SDKMAN Available in Non-Interactive Shells

Add the SDKMAN init line to ~/.bashrc so Java is on PATH after every login:

# This is usually added automatically by the SDKMAN installer,
# but verify it's in .bashrc:
grep -q 'sdkman-init' ~/.bashrc || \
  echo 'source "$HOME/.sdkman/bin/sdkman-init.sh"' >> ~/.bashrc

source ~/.bashrc
java -version

Install Maven

# Install Maven via SDKMAN (keeps version management consistent)
sdk install maven

# Verify
mvn -version

Install Git

# Usually pre-installed on Ubuntu 22.04, but verify:
git --version

# If not installed:
sudo apt-get install -y git

Build the Custom JDK Base Image (therapy-jdk25)

All 8 Spring Boot Dockerfiles start with FROM therapy-jdk25. This is a custom base image that packages JDK 25 into a Docker image. It must be built once on the Oracle VM before any service images can be built.

Create the Base Image Dockerfile

Create a file ~/therapy-jdk25/Dockerfile with the following content:

~/therapy-jdk25/Dockerfile
FROM ubuntu:22.04

# Suppress interactive prompts during apt installs
ENV DEBIAN_FRONTEND=noninteractive

# Install wget, curl, and fontconfig (needed by some JVM internals)
RUN apt-get update && \
    apt-get install -y wget curl fontconfig ca-certificates && \
    rm -rf /var/lib/apt/lists/*

# Download and install Oracle OpenJDK 25
# Check https://jdk.java.net/25/ for the latest build URL
ARG JDK_URL=https://download.java.net/java/early_access/jdk25/1/GPL/openjdk-25-ea+1_linux-aarch64_bin.tar.gz
RUN mkdir -p /opt/java && \
    wget -q -O /tmp/jdk.tar.gz "${JDK_URL}" && \
    tar -xzf /tmp/jdk.tar.gz -C /opt/java --strip-components=1 && \
    rm /tmp/jdk.tar.gz

ENV JAVA_HOME=/opt/java
ENV PATH="$JAVA_HOME/bin:$PATH"

# Verify Java installed correctly
RUN java -version
Architecture note: The ARM VM requires the linux-aarch64 build. If you are on an x86 VM, use the linux-x64 build URL instead. Check jdk.java.net/25 for the latest download URL for your architecture.

Build the Base Image

# Build from the directory containing the Dockerfile
cd ~/therapy-jdk25
docker build -t therapy-jdk25 .

# Verify the image exists
docker images | grep therapy-jdk25

# Quick sanity check
docker run --rm therapy-jdk25 java -version

This base image only needs to be built once per machine. All subsequent service image builds inherit from it.

Generate JWT RSA Key Pair

All services use RSA-based JWT tokens. The Gateway signs tokens using the private key and verifies them using the public key. All other services receive only the public key to verify tokens — they cannot issue them.

Generate the Keys

# Make sure you are in the repository root
cd ~/repository

# Create the keys directory
mkdir -p keys

# Generate a 4096-bit RSA private key
openssl genrsa -out keys/gateway-private.pem 4096

# Extract the public key from the private key
openssl rsa -in keys/gateway-private.pem -pubout -out keys/gateway-public.pem

# Verify both files exist and look correct
ls -la keys/
cat keys/gateway-public.pem
Keep the private key secret. Never commit gateway-private.pem to Git. Verify your .gitignore has keys/ listed. The public key can be committed if needed, but the private key must never leave the server.

How Keys Are Used

In docker-compose.yml, the ./keys directory is mounted as a read-only volume into every service container at /app/keys/. The GatewayService gets both keys; all other services get only the public key:

# Gateway (application.properties)
jwt.private-key=classpath:keys/gateway-private.pem
jwt.public-key=classpath:keys/gateway-public.pem

# All other services (application.properties)
jwt.public-key=classpath:keys/gateway-public.pem

Google Cloud Console — OAuth 2.0 Setup & Refresh Token

Two services use Google APIs: UserService (sends email via Gmail API) and NotificationService (creates/updates Google Calendar events). Both use the same OAuth 2.0 credentials — a single set of Client ID, Client Secret, and Refresh Token.

10.1 Create a Google Cloud Project

  1. Go to console.cloud.google.com. Sign in with the Google account you want to send mail from / create calendar events on.
  2. Click the project dropdown at the top → New Project. Name it something like therapy-connect-api.
  3. Select the new project once created.

10.2 Enable Required APIs

  1. Go to APIs & Services → Library.
  2. Search for and enable Gmail API → click Enable.
  3. Search for and enable Google Calendar API → click Enable.

10.3 Configure OAuth Consent Screen

  1. Go to APIs & Services → OAuth consent screen.
  2. Choose External (unless you are a Google Workspace org).
  3. Fill in App name (e.g. Therapy Connect), support email, and developer contact email.
  4. On the Scopes page, add these scopes:
    • https://www.googleapis.com/auth/gmail.send
    • https://www.googleapis.com/auth/calendar
  5. On the Test users page, add the email address you want to use (the therapist's Gmail address).
  6. Save and continue through to the Summary.
Why External + Test Users? For a personal/small-scale app, staying in "Testing" mode is fine. The refresh token won't expire unless the app goes unused for 6 months. Publishing the app requires Google verification for sensitive scopes.

10.4 Create OAuth 2.0 Credentials

  1. Go to APIs & Services → Credentials → Create Credentials → OAuth client ID.
  2. Application type: Web application.
  3. Name: therapy-connect-backend.
  4. Under Authorized redirect URIs, add: https://developers.google.com/oauthplayground
  5. Click Create. You will see your Client ID and Client Secret — save both.

10.5 Get the Refresh Token via OAuth Playground

  1. Go to developers.google.com/oauthplayground.
  2. Click the gear icon (⚙️) in the top right → check Use your own OAuth credentials.
  3. Paste in your Client ID and Client Secret from step 10.4.
  4. In the left panel, find and select these scopes:
    • Gmail API → https://www.googleapis.com/auth/gmail.send
    • Google Calendar API → https://www.googleapis.com/auth/calendar
  5. Click Authorize APIs. Google will prompt you to sign in and grant permissions — sign in as the therapist's Google account.
  6. After authorization, click Exchange authorization code for tokens.
  7. The response panel shows the Refresh token — copy and save it. This is your GOOGLE_REFRESH_TOKEN.
Important: The refresh token is long-lived but can be invalidated if: (a) you revoke access in Google account settings, (b) the OAuth consent screen is deleted, or (c) the app goes unused for 6+ months in testing mode. If calendar/email stops working, generate a new refresh token using these same steps.

Configure the .env File

The docker-compose.yml reads environment variables from a .env file in the repository root. This file must never be committed to Git — it contains database passwords and API secrets.

Create .env from the Template

cd ~/repository
cp .env.example .env
nano .env

Fill In All Values

~/repository/.env
# ── Database credentials ─────────────────────────────────────────────────────
DB_USERNAME=postgres
DB_PASSWORD=your_strong_password_here     # choose something secure, not "changeme"

# ── Google OAuth 2.0 ─────────────────────────────────────────────────────────
GOOGLE_CLIENT_ID=xxxxxxxx.apps.googleusercontent.com
GOOGLE_CLIENT_SECRET=GOCSPX-xxxxxxxxxxxxxxxxxxxx
GOOGLE_REFRESH_TOKEN=1//xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx

# ── Email settings ───────────────────────────────────────────────────────────
THERAPIST_EMAIL=therapist@gmail.com       # the Google account used for Calendar
GMAIL_FROM=noreply@yourdomain.com         # the From address for sent emails

# ── App secrets ──────────────────────────────────────────────────────────────
APP_ENCRYPTION_SECRET=your_32_char_aes_key_here   # used by TherapistService for encryption

# ── Frontend URL (used for CORS and email links) ──────────────────────────────
FRONTEND_URL=https://your-vercel-app.vercel.app   # update after Vercel deploy
APP_ENCRYPTION_SECRET: must be exactly 32 characters for AES-256. Generate one with: openssl rand -base64 24 | tr -d '=/+' | head -c 32
FRONTEND_URL: set this to your Vercel app URL. It's used by UserService to generate correct links in emails, and by the Gateway for CORS. If you don't have the Vercel URL yet, you can set a placeholder and update it later — just restart the services after changing.

Docker Compose — Build Images & Start All Services

With JARs built and the .env file in place, Docker Compose can build and start everything. On the first run, this takes 5–10 minutes as Docker downloads base images and builds all service images.

Understanding the Startup Order

The depends_on conditions in docker-compose.yml enforce this order:

  1. postgres and kafka start first (they have no dependencies)
  2. All Spring Boot services wait until postgres and kafka pass their healthcheck
  3. discovery (Eureka) also starts early
  4. All other services wait until discovery is healthy too

First-Time Build and Start

cd ~/repository

# Build all images and start all services in detached mode
docker compose up --build -d

# Watch the logs to see services come up
docker compose logs -f
What happens to the database? On the very first start, PostgreSQL initializes and runs every SQL file in postgres-init/. This creates the 7 databases (gateway, user, therapist, client, appointment, notification, analytics). Spring Boot then runs Hibernate/Flyway to create the tables. This all happens automatically — you do not need to manually create anything.

Check All Services Are Running

# See status of all containers
docker compose ps

# Check logs for a specific service
docker compose logs -f gateway
docker compose logs -f therapist

All containers should show status Up or Up (healthy). If a service keeps restarting, check its logs for the error.

Common First-Run Issues

SymptomLikely CauseFix
Service exits with Connection refused to postgresPostgres not yet healthy when service startsWait a minute and run docker compose restart <service>
Kafka consumer silently drops all messagestrusted.packages is too restrictiveSet spring.json.trusted.packages=* in application.properties for all consumer services
therapy-jdk25 not found during buildBase image not built yetRun Section 8 steps first
Service logs show PKIX path building failedSSL cert issue in a containerUsually a transient DNS issue — restart the container

Stop and Start Commands

# Stop all services (preserves volumes/data)
docker compose down

# Start again without rebuilding
docker compose up -d

# Stop and remove all data (nuclear option — deletes all databases)
docker compose down -v
Never run docker compose down -v on a production system — the -v flag deletes the named volumes postgres_data and kafka_data, permanently destroying all data.

DuckDNS Setup — Dynamic DNS for Oracle VM

Oracle Free Tier VMs can have their public IP change if the instance is terminated and recreated. DuckDNS provides a free dynamic DNS hostname that you update automatically via a cron job — so your Vercel frontend always points to the right IP even if it changes.

14.1 Create a DuckDNS Account and Subdomain

  1. Go to duckdns.org and sign in with any of the supported providers (GitHub, Reddit, Google, etc.).
  2. In the domains section, type a subdomain name (e.g. therapy-connect) and click Add domain.
  3. You now have therapy-connect.duckdns.org pointing to no IP yet.
  4. Note your token shown at the top of the page — it's a UUID that authenticates your DNS update requests.

14.2 Set Up Auto-Update on the Oracle VM

Create a script that calls the DuckDNS API to update the IP, then schedule it to run every 5 minutes via cron.

# Create a directory for the DuckDNS script
mkdir -p ~/duckdns

# Create the update script
cat > ~/duckdns/duck.sh <<'EOF'
#!/bin/bash
echo url="https://www.duckdns.org/update?domains=YOUR_SUBDOMAIN&token=YOUR_TOKEN&ip=" \
  | curl -k -o ~/duckdns/duck.log -K -
EOF

# Replace placeholders with your actual values
nano ~/duckdns/duck.sh

# Make it executable
chmod 700 ~/duckdns/duck.sh

# Test it — should output OK
~/duckdns/duck.sh
cat ~/duckdns/duck.log

The log file should contain OK if the update succeeded, or KO if the token or subdomain is wrong.

14.3 Schedule with Cron

# Open crontab editor
crontab -e

# Add this line to run the update every 5 minutes:
*/5 * * * * ~/duckdns/duck.sh >/dev/null 2>&1

14.4 Verify the DNS Points to Your VM

# From your local machine — should return your Oracle VM's public IP
nslookup YOUR_SUBDOMAIN.duckdns.org

# And confirm the gateway is reachable via the hostname
curl http://YOUR_SUBDOMAIN.duckdns.org:8091/actuator/health

After this step, your Vercel frontend should use the DuckDNS hostname as the API base URL. Update FRONTEND_URL in your .env if needed and restart the services: docker compose up -d.

Vercel — Deploy the React Frontend

The React frontend lives in auth-app/ inside the repository and is built and deployed automatically by Vercel whenever code is pushed to GitHub. You do not build or host it on the Oracle VM.

15.1 Connect the Repository to Vercel

  1. Go to vercel.com and sign in (use your GitHub account for the easiest setup).
  2. Click Add New → Project.
  3. Select your GitHub repository from the list. Click Import.
  4. Vercel will auto-detect it as a Vite project. Configure the build settings:
    SettingValue
    Framework presetVite
    Root Directoryauth-app
    Build commandnpm run build (auto-detected)
    Output directorydist (auto-detected)
  5. Important: Set the Root Directory to auth-app since the frontend is in a subdirectory, not at the repository root.

15.2 Set Environment Variables in Vercel

  1. Before clicking Deploy, go to the Environment Variables section.
  2. Add the following variable:
    NameValue
    VITE_API_BASE_URLhttp://YOUR_SUBDOMAIN.duckdns.org:8091
  3. This variable tells the React app where to send API requests. It points to your Oracle VM's gateway via DuckDNS.
  4. Click Deploy.
HTTP vs HTTPS: If your Oracle VM does not have an SSL certificate set up (no Nginx + Certbot), the API URL uses plain HTTP. Modern browsers may block mixed-content requests (HTTPS frontend → HTTP backend). You have two options: (a) set up Nginx + Let's Encrypt on the Oracle VM for HTTPS (recommended for production), or (b) use a browser that allows mixed content during development.

15.3 Automatic Deploys

After the initial setup, every push to the connected GitHub branch triggers an automatic Vercel build and deploy. No action needed — Vercel rebuilds the frontend automatically.

15.4 Update FRONTEND_URL in .env

After the first Vercel deploy, you get a URL like https://your-app.vercel.app. Update this in the backend .env:

# On the Oracle VM, in ~/repository/.env:
FRONTEND_URL=https://your-app.vercel.app

# Restart the services to pick up the new env var
docker compose up -d

This URL is used by UserService to generate correct links in verification emails, and by the Gateway's CORS configuration to allow requests from the frontend.

15.5 Update Environment Variable in Vercel (if URL changes)

If you ever redeploy and the Vercel URL changes, or if the Oracle VM IP changes and you update the DuckDNS record:

  1. In Vercel → Project → Settings → Environment Variables.
  2. Edit VITE_API_BASE_URL with the new value.
  3. Go to Deployments → click the latest deploy → Redeploy (with existing build cache is fine).

Day-to-Day Operations — Update & Rebuild Workflow

This section covers the commands you'll run most often once the system is live.

Deploying a Code Update

When you push a code change and want to deploy it to the Oracle VM:

cd ~/repository

# Step 1: Pull the latest code from GitHub
git pull origin main

# Step 2: Re-build the JAR for the service(s) that changed
# If only TherapistService changed:
mvn -f TherapistService/pom.xml clean package -DskipTests

# If multiple services changed, or you're not sure, build all:
mvn clean package -DskipTests

# Step 3: Rebuild the Docker image and restart only the changed service
docker compose build therapist
docker compose up -d therapist

# OR rebuild and restart all services at once:
docker compose up --build -d
Targeted rebuilds save time. If only one service changed, only rebuild that one. docker compose build therapist && docker compose up -d therapist takes ~30 seconds vs rebuilding all 8 services.

Restarting a Single Service

# Restart without rebuilding the image
docker compose restart gateway

# Or stop and start
docker compose stop notification
docker compose start notification

Updating Environment Variables

# Edit the .env file
nano ~/repository/.env

# Restart the services that use the changed variable
# (all services re-read .env on start, not hot-reload)
docker compose up -d

Viewing Resource Usage

# CPU and memory usage per container (live)
docker stats

# Disk usage by Docker
docker system df

Cleaning Up Old Docker Images

# Remove dangling images (untagged leftovers from old builds)
docker image prune -f

# Remove all unused images, containers, networks (more aggressive)
docker system prune -f

Backup PostgreSQL Data

# Dump a specific database (e.g. therapist)
docker exec tc-postgres pg_dump -U postgres therapist > ~/backup_therapist_$(date +%Y%m%d).sql

# Dump all databases
docker exec tc-postgres pg_dumpall -U postgres > ~/backup_all_$(date +%Y%m%d).sql

# Restore a dump
docker exec -i tc-postgres psql -U postgres -d therapist < ~/backup_therapist_20250101.sql

Handling VM Reboots

Docker is configured to auto-start (sudo systemctl enable docker). All containers have restart: unless-stopped in docker-compose.yml. After a VM reboot:

# Services should auto-start. Verify:
docker compose ps

# If they didn't start automatically:
cd ~/repository && docker compose up -d
