Google OAuth Setup Guide

One-time configuration to enable calendar appointment invites and password reset emails using a single Google account. Estimated time: 20–30 minutes.

Contents

  1. Create a Google Cloud Project
  2. Enable the APIs (Calendar + Gmail)
  3. Configure the OAuth Consent Screen
  4. Create OAuth 2.0 Credentials
  5. Get a Refresh Token via OAuth Playground
  6. Set Environment Variables
What you will end up with: Three values — GOOGLE_CLIENT_ID, GOOGLE_CLIENT_SECRET, and GOOGLE_REFRESH_TOKEN — plus the Gmail address you authorise. These are set as environment variables on the server and never need to be changed unless you revoke access or rotate credentials intentionally.
Which Google account to use: Use a dedicated platform account (e.g. yourpractice.noreply@gmail.com), not your personal account. Calendar invites will be sent from this account and so will password reset emails. Clients and therapists will see this address as the sender.

1 Create a Google Cloud Project

1
Open Google Cloud Console

Go to console.cloud.google.com and sign in with the Google account that will send emails and calendar invites (the platform account, not your personal one).

2
Create a new project
  1. Click the project selector at the very top of the page (it shows "Select a project" or an existing project name).
  2. In the pop-up, click New Project (top right of the dialog).
  3. Enter a project name such as Therapy Platform.
  4. Leave Organisation as-is and click Create.
  5. Wait a few seconds, then click the notification bell or the project selector again and select your new project.
You should now see "Therapy Platform" in the top bar where the project selector is.

2 Enable the APIs

3
Enable the Google Calendar API
APIs & Services›Library
  1. In the left sidebar, click APIs & Services, then Library.
  2. Search for Google Calendar API.
  3. Click the result and then click Enable.
The button changes to "Manage" once enabled. That means it worked.
4
Enable the Gmail API
APIs & Services›Library
  1. Go back to Library.
  2. Search for Gmail API.
  3. Click the result and click Enable.
Both APIs are now enabled. You can verify by going to APIs & Services → Enabled APIs & services and confirming both appear in the list.

3 Configure the OAuth Consent Screen

This screen is what users see when they are asked to grant permission to the platform. You will configure it once. Google has recently updated this UI — the steps below match the current layout.

5
Open the OAuth Consent Screen (now called "Google Auth Platform")
APIs & Services›OAuth consent screen

In the left sidebar click APIs & Services, then OAuth consent screen. You will land on the Google Auth Platform page with a left-side menu showing: Branding, Audience, Clients, Data access, Verification centre, Settings.

If you see a "Get started" button instead of the menu, click it and complete the initial setup wizard first.
6
Branding — fill in app details
Branding
  1. App name: Therapy Platform
  2. User support email: your email address
  3. Developer contact email: your email address
  4. Click Save.

Logo and homepage URL are optional — leave them blank for now.

7
Audience — set user type and add test user
Audience
  1. Click Audience in the left menu.
  2. Under "User type", select External and save.
  3. Scroll down to the Test users section.
  4. Click Add users and add the Gmail address of the platform account (the same account you are signed in with — the one that will send emails and calendar invites).
  5. Click Save.
The app stays in "Testing" mode which is fine. Only accounts listed as test users can authorise the app. Since only the platform account needs to authorise it, one test user is all you need. You do NOT need to submit the app for Google verification.
8
Data access — add both OAuth scopes
Data access
  1. Click Data access in the left menu.
  2. Click Add or remove scopes.
  3. In the filter/search box, search for calendar and tick: https://www.googleapis.com/auth/calendar
  4. Search for gmail and tick: https://www.googleapis.com/auth/gmail.send
  5. Click Update, then Save.
Make sure you add scopes under Data access, not anywhere else. The redirect URI for OAuth Playground (added in the next phase) goes under Clients — these are two completely different fields.
After saving, both scopes should appear in the "Your non-sensitive scopes" or "Your sensitive scopes" list on the Data access page.

4 Create OAuth 2.0 Credentials

9
Create a Web Application OAuth client
Clients
  1. Click Clients in the left menu.
  2. Click Create client.
  3. Application type: select Web application.
    Do not select "Desktop app". Desktop apps cannot have external redirect URIs, which means you cannot use the OAuth Playground with them.
  4. Name: Therapy Platform Web Client (any name is fine).
  5. Under Authorised redirect URIs, click Add URI and enter exactly: 
    https://developers.google.com/oauthplayground
  6. Click Create.
10
Copy your Client ID and Client Secret

After creating the client, a dialog appears showing your credentials. Copy and save both values securely — you will need them in the next phase.

Values to save now:

Client ID — looks like: 123456789-abcdefg.apps.googleusercontent.com
Client Secret — looks like: GOCSPX-xxxxxxxxxxxxxxxxxxxxxx
You can always retrieve these later by going to Clients, clicking the client name, and scrolling to the "Client secrets" section.

5 Get a Refresh Token via OAuth Playground

The OAuth Playground is a tool provided by Google that lets you generate OAuth tokens without writing any code. You will use it once to get a refresh token — a long-lived credential the server uses to send calendar invites and emails on behalf of the platform account.

11
Open OAuth Playground in a private / incognito window

Open a private/incognito browser window and go to:

https://developers.google.com/oauthplayground
Using an incognito window avoids Google defaulting to the wrong signed-in account. Make sure you sign in as the platform account (not your personal Gmail) when prompted.
12
Enter your own OAuth credentials
  1. Click the gear icon (⚙) in the top-right corner of the page.
  2. Tick "Use your own OAuth credentials".
  3. Enter the Client ID and Client Secret you copied in step 10.
  4. Close the settings panel (click elsewhere or press Escape).
The gear panel should close and your client ID should be visible in the settings area.
13
Select both API scopes

In the left panel under "Select & authorize APIs", find and tick both of the following scopes:

  1. Expand Google Calendar API v3 and tick:
    https://www.googleapis.com/auth/calendar
  2. Expand Gmail API v1 and tick:
    https://www.googleapis.com/auth/gmail.send
You can also manually type or paste the scope URLs into the input box at the bottom of the left panel, separated by a space, then press Enter.
You must select both scopes. If you only select one, the server will be able to do calendar invites or emails, but not both.
14
Authorise the APIs
  1. Click Authorize APIs.
  2. A Google sign-in page opens. Sign in as the platform account (the account you added as a test user in step 7).
  3. Google will warn that the app is unverified — click Continue / Advanced → Go to Therapy Platform (unsafe). This is expected for an app in testing mode.
  4. Google shows the consent screen listing the Calendar and Gmail permissions being requested. Review them and click Allow to grant access.
You will be redirected back to the OAuth Playground. Step 2 in the right panel ("Exchange authorization code for tokens") becomes active.
15
Exchange code for tokens and copy the refresh token
  1. Click Exchange authorization code for tokens.
  2. The response panel on the right shows a JSON response containing access_token, refresh_token, and expires_in.
  3. Copy the value of "refresh_token" — it starts with 1// and is a long string.

Value to save now:

Refresh Token — looks like: 1//04xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
Do not close this tab yet. The refresh token is only shown once in this flow. If you lose it, you can repeat steps 11–15 to get a new one — it does not break anything.
There is an "Auto-refresh the token before it expires" checkbox on the page — you can leave it in any state. The server handles token refresh automatically using the refresh token; the access token shown here is not used.

6 Set Environment Variables

Set the following environment variables on the server where the application runs. The exact method depends on your deployment (Docker Compose, system environment, secrets manager, etc.) — ask your developer if unsure where to set them.

Environment Variable Status Value
GOOGLE_CLIENT_ID New Client ID from step 10
e.g. 123456789-abcdefg.apps.googleusercontent.com
GOOGLE_CLIENT_SECRET New Client Secret from step 10
e.g. GOCSPX-xxxxxxxxxxxxxxxxxxxx
GOOGLE_REFRESH_TOKEN New Refresh token from step 15
e.g. 1//04xxxxxxxxxxxxxxxxxxxx
THERAPIST_EMAIL New The therapist's email address — they will receive calendar appointment invites.
e.g. dr.smith@example.com
GMAIL_FROM New The platform Gmail address — this appears as the "From" address on password reset emails.
Must be the same account you signed in as during step 14.
e.g. yourpractice.noreply@gmail.com
FRONTEND_URL New The base URL of the frontend application — used in the password reset link sent to users.
e.g. https://yourplatform.com   or   http://localhost:5173 for local dev
Once these six variables are set and both services are restarted, the setup is complete.
NotificationService will use GOOGLE_CLIENT_ID, GOOGLE_CLIENT_SECRET, GOOGLE_REFRESH_TOKEN, and THERAPIST_EMAIL for calendar invites.
UserService will use the same three GOOGLE_* variables plus GMAIL_FROM and FRONTEND_URL for password reset emails.

? Troubleshooting

!
Error: redirect_uri_mismatch

The redirect URI in your OAuth client does not include the OAuth Playground URL. Go back to step 9, open the client you created, and confirm https://developers.google.com/oauthplayground is listed under Authorised redirect URIs. Save and retry step 11.

!
Error: Access blocked — this app's request is invalid

You may have selected "Desktop app" instead of "Web application" when creating the OAuth client. Desktop apps do not support the OAuth Playground redirect URI. Create a new client of type Web application (step 9) and use its credentials.

!
Error: This account cannot be used — not a test user

The Google account you signed in with during step 14 is not in the test users list. Go to Audience → Test users → Add the account email → Save, then retry step 14.

!
Calendar invites work but password reset emails do not (or vice versa)

The refresh token was generated with only one scope. Repeat steps 11–15, making sure both scopes are ticked before clicking "Authorize APIs". The new refresh token replaces the old GOOGLE_REFRESH_TOKEN value.

!
The refresh token stopped working after a long period

Refresh tokens for apps in "Testing" mode expire after 7 days if the app has not been verified by Google. To avoid this:

  • Short term: repeat steps 11–15 to get a new refresh token whenever it expires.
  • Long term: publish the app (submit for Google verification) to get non-expiring tokens. For an internal platform this is usually not required — ask your developer.
Apps in testing mode that are used by fewer than 100 users can remain in testing indefinitely; only the refresh tokens expire every 7 days. If your developer changes the OAuth client to "Internal" user type (requires a Google Workspace account), tokens do not expire.
!
Gmail scope does not appear in the Data access list

Make sure the Gmail API is enabled (step 4). Google only shows scopes for APIs that are enabled. Enable the Gmail API first, then go back to Data access and add the Gmail scope.

Security note: Treat GOOGLE_CLIENT_SECRET and GOOGLE_REFRESH_TOKEN as passwords. Store them in a secrets manager or encrypted environment — never commit them to source control or include them in application config files that are checked into git.
